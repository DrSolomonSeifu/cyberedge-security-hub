4A

Topic 4A — Authentication

Authentication is the process of verifying that a subject — a user or device — is genuinely who or what it claims to be. It is the gateway to every system, resource, and service your organization depends on.

4A Authentication Design

Authentication Design Framework

Authentication happens when a supplicant (the user or device claiming an identity) presents credentials to an authentication server. The server compares the submitted credentials against stored records. Designing this exchange securely requires satisfying all three pillars of the CIA triad.

CONFIDENTIALITY

Credentials must be protected during storage and transmission. If a password database stores plaintext or an exchange is unencrypted, confidentiality has failed — and any attacker who gains access inherits the account holder's full privileges.

INTEGRITY

The authentication mechanism must resist forgery and manipulation. Counterfeit credentials, replayed tokens, or spoofed identities must not be accepted. An authentication system defeated by impersonation has failed regardless of how complex the credentials are.

AVAILABILITY

Authentication must not block legitimate users. Overly complex or unreliable mechanisms that lock users out or generate excessive help-desk calls are availability failures — and users who find them too burdensome will circumvent them, creating new vulnerabilities.

The knowledge factor is the oldest category. A password combines with a username: the username identifies who is claiming access; the password is the secret known only to the account holder. A passphrase (multiple words strung together) offers better security and usability than a short password. A PIN is device-bound — valid only on the device it was configured on, making it useless to an attacker even if stolen.
Knowledge FactorDescriptionKey Characteristic
PasswordSecret string known only to the account holderMust never be shared or reused
PassphraseMultiple words combined — longer than a passwordMore secure AND more memorable
PINNumeric or alphanumeric codeBound to a single device only
4A Password Concepts

Password Concepts & Policy

Passwords remain the most widely deployed authentication mechanism — and credential mismanagement remains one of the most exploited attack vectors. Without enforceable policy, users default to weak, reused passwords that put the entire organization at risk.

📋 SYSTEM-ENFORCED ACCOUNT POLICIES
  • Length — Minimum (and sometimes maximum) character count. Length is the single greatest contributor to password strength, dramatically increasing brute-force search space.
  • Complexity — Combination of uppercase, lowercase, numbers, and symbols. Also prohibits embedding the username in the password.
  • Aging vs. Expiration — Aging: user can still log in but must change on next login. Expiration: account is locked until a new password is set.
  • History & Minimum Age — History blocks reuse of N previous passwords. Minimum age prevents users from rapidly cycling through changes to restore their preferred password.
Soft policy risk — credential stuffing: when users reuse work passwords on external sites (retail, social media), a breach of that external site gives attackers direct access to your enterprise network. This cannot be technically enforced — only addressed through training.
NIST now deprecates three traditionally recommended practices: (1) mandatory complexity rules — they create predictable patterns like Password1!; (2) periodic forced password changes — unless there is evidence of compromise, forced rotation produces weaker passwords; (3) password hints — they reveal too much information. NIST's current guidance prioritizes length + uniqueness + password managers.
4A Password Managers

Password Managers

The practical answer to the password reuse problem is the password manager — a tool that generates, stores, and auto-fills unique strong passwords for every account, so the user only needs to remember one master credential.

1

Choose a Password Manager

Built-in OS/browser options (Windows Credential Manager, Apple iCloud Keychain) or third-party browser plug-ins (LastPass, 1Password, Bitwarden). Both types follow the same vault model.

2

Secure the Vault with a Master Password

The vault is an encrypted container holding all stored credentials. The master password is the single key — the one credential the user must protect above all others. Vaults are typically cloud-synced for multi-device access.

3

Generate Random Passwords

When creating or updating accounts, the manager generates a random password meeting the site's length and complexity requirements. The user never sees or memorizes it.

4

Validate & Auto-Fill

On subsequent visits, the manager validates the site's digital certificate before offering to auto-fill. Certificate validation is the critical defense against phishing — it prevents the manager from filling credentials into spoofed sites.

RISK 1

Weak master password — the vault is only as secure as the master password protecting it. A weak or reused master password compromises every stored credential.

RISK 2

Vendor cloud breach — if the password manager provider is compromised, stored vaults may be exposed. The LastPass breach is a real-world example of this risk.

RISK 3

Impersonation attacks — spoofed sites may attempt to trigger auto-fill. The certificate validation step defends against this, but only if the manager performs that check reliably.

4A Multifactor Authentication

Multifactor Authentication — MFA

A single knowledge factor — a password — is insufficient for securing modern accounts. MFA strengthens authentication by requiring credentials from more than one category of factor. The exam tests this distinction carefully.

MFA requires credentials from different factor categories — not multiple items from the same category. A PIN plus a date of birth is NOT MFA — both are "something you know." True MFA combines, for example, something you know WITH something you have.
🧠
SOMETHING YOU KNOW
Knowledge Factor

A secret held in memory. The most common factor — but the weakest, because secrets can be guessed, stolen, or socially engineered.

Password PIN Passphrase
📲
SOMETHING YOU HAVE
Ownership Factor

A physical object only the account holder possesses. Even if a password is stolen, the attacker cannot authenticate without the physical token.

Smart card Key fob Phone
👁️
SOMETHING YOU ARE
Biometric Factor

A physical or behavioral characteristic unique to the individual. Cannot be forgotten or transferred, but carries enrollment and accuracy challenges.

Fingerprint Face scan Gait
📍
SOMEWHERE YOU ARE
Location Factor

A geographic or network location element. Used as a continuous authentication control on top of primary factors — not as a standalone primary factor.

GPS IP address Network port
📌 KEY DISTINCTIONS
  • 2FA (Two-Factor Authentication) — a specific instance of MFA using exactly two factors. All 2FA is MFA, but MFA is not limited to two factors.
  • Location factor — not a standalone primary factor. Used as a continuous monitoring mechanism or an additional layer on top of other factors.
  • Impossible travel — a behavioral signal where login events from two geographically distant locations occur within an impossible timeframe. Many platforms auto-flag and disable these accounts.
4A Biometric Authentication

Biometric Authentication

Biometrics verify identity using physical or behavioral characteristics unique to the individual. Two stages define the system: enrollment (capturing and storing the template) and verification (comparing a new scan to the stored template within a tolerance threshold).

SENSOR MODULE

Captures the biometric sample — fingerprint scanner, camera, infrared sensor, etc. Quality of the sensor directly impacts accuracy.

FEATURE EXTRACTION

Converts the raw scan into a mathematical template — an abstraction of the distinguishing characteristics. The template, not the original image, is stored.

TOLERANCE THRESHOLD

Because biometric scans are never a perfect match (different angles, lighting, sensor variation), the system accepts matches within a defined tolerance. This threshold drives the error rates below.

MetricAlso CalledWhat It MeasuresSecurity Impact
FRRType I Error / False Non-Match RateLegitimate user rejectedInconvenience — no security breach
FARType II Error / False Match RateIntruder acceptedSecurity breach — highest priority to reduce
CERCrossover Error RatePoint where FRR = FARLower CER = more accurate technology
FERFailure to Enroll RateCannot create a usable templateAccessibility — some users cannot enroll
When tuning a biometric system, reducing FAR (false acceptances) is always the higher security priority — a false acceptance means an intruder gained access. The CER is the reference point for comparing biometric technologies: lower CER = better overall accuracy.
FINGERPRINT RECOGNITION

Most widely deployed biometric. Inexpensive capacitive or optical sensors. Non-intrusive and user-accepted. Limitation: moisture, dirt, or cuts can prevent accurate reads.

FACIAL RECOGNITION

Records spatial measurements of facial features. Modern systems combine optical + infrared cameras. The infrared component defeats photo/print spoofing — a 2D image does not reflect IR the same way a live face does.

4A Hard Authentication Tokens

Hard Authentication Tokens

Hard tokens are physical devices — authenticators — that the account holder possesses and operates. The token is generated within a secure cryptoprocessor and is never transmitted in a way that can be intercepted and reused. Three generation methods exist.

CERT-BASED

Certificate-Based Auth

The supplicant's private key generates a unique signed token. The identity provider verifies it using the corresponding public key — the PKI model from Lesson 3 applied to authentication.

⚠ Requires full PKI infrastructure — resource-intensive to deploy and maintain.

OTP

One-Time Password

A hash of a shared secret combined with a synchronization seed. TOTP uses a timestamp (changes every 30s). HOTP uses a counter (changes each event). Each token is single-use.

⚠ Both device and server share the same secret — if that secret is stolen, the entire scheme is broken.

FIDO U2F

FIDO Universal 2nd Factor

Uses a public/private key pair per account. The private key never leaves the device. The public key registers with the server. No shared secret — eliminates the HOTP/TOTP weakness.

✓ PKI used only for device attestation, not user auth directly.

Form FactorHow It WorksConnection Type
Smart CardStores digital certificate, private key, and PIN. PIN activates the card — two factors in one device.Contact reader or NFC (contactless)
OTP Key FobCryptoprocessor generates time-based or counter-based one-time code. User reads and enters the displayed code.No computer connection needed
Security KeyPortable HSM — USB or NFC. Most closely associated with FIDO U2F. Requires presence activation (button or biometric fingerprint). PIN as backup.USB or NFC
Static tokens — physical devices that transmit a fixed code — are highly vulnerable to cloning and replay attacks. Many building entry systems still use them. Dynamic tokens (OTP, cert-based, FIDO U2F) exist specifically to defeat replay. Know this distinction for the exam.
4A Soft Authentication Tokens

Soft Authentication Tokens

Soft tokens deliver one-time credentials through software and existing communication channels rather than dedicated hardware devices. The term out-of-band describes delivery through a channel different from the one being used to authenticate.

SMS

Code sent as a text message to the registered phone number. Convenient but vulnerable to SIM-swapping attacks and network interception.

EMAIL

Code delivered to a registered email address. Shares the same weaknesses as SMS — email account compromise transfers the vulnerability.

PUSH NOTIFICATION

A prompt sent directly to an authenticator app on the user's registered device — showing location, time, and account details so the user can recognize a legitimate vs. fraudulent request.

AUTHENTICATOR APP

Generates OTP tokens locally on the device — never transmitted over an interceptable channel. Must be unlocked with the device credential (PIN, fingerprint, face). Most secure soft token option.

Critical exam distinction — SMS and email-based codes are called two-step verification, NOT true MFA. They are delivered to accounts/channels accessible from multiple devices and are vulnerable to interception. They are better than a password alone, but weaker than genuine MFA. An authenticator app qualifies as a true ownership factor because the token is generated locally and the device credential adds a second layer.
An authenticator app runs on a shared-use smartphone that also runs other applications. This creates a risk: malware installed on that device could potentially compromise the app or intercept the token at the point of display. This is the key limitation distinguishing soft tokens from dedicated hard token devices, which run no other software.
4A Passwordless Authentication

Passwordless Authentication — FIDO2 / WebAuthn

Passwordless authentication eliminates the password entirely — it is not a backup, not a fallback, not present in the workflow at all. The framework that enables this is FIDO2 with WebAuthn. The core innovation: credentials are validated locally on the device and never sent to any server.

1

Choose an Authenticator

Roaming — portable security key (used across devices). Platform — built into the OS (Windows Hello, Face ID, Touch ID).

2

Configure a Local Gesture

Fingerprint, face recognition, or PIN — validated only by the authenticator itself, never sent to any server. The server only ever receives the cryptographic result of a successful local verification.

3

Register with the Relying Party

For each service, the authenticator generates a unique public/private key pair. The public key is registered with the relying party. The private key never leaves the device.

4

Authenticate via Challenge-Response

The user performs the local gesture → unlocks the private key → the private key signs the challenge → the signed response is sent to the relying party.

5

Relying Party Verifies

The stored public key verifies the signature. If valid — access is granted. No password. No shared secret crossing the network.

ATTESTATION

The mechanism by which an authenticator proves it is a legitimate, genuine device — not a clone or spoof. Each security key carries an attestation key identifying its brand and model.

PRIVACY DESIGN

Attestation keys identify a model, not an individual device. If they were device-unique, they could be used to track users across services — a serious privacy violation by design.

FIDO2 vs U2F

FIDO2/WebAuthn adds an API for fully password-free web authentication — something FIDO U2F alone could not achieve. Most FIDO U2F keys also support FIDO2.

QUESTION 1 OF 3 · AUTHENTICATION
A company requires employees to enter both a password and a code sent to their personal email address before accessing the VPN. A security auditor flags this as insufficient for a system storing regulated healthcare data. Why is this system NOT considered true MFA?
ABecause the password and the email code are both delivered over the internet
BBecause both credentials belong to the same factor category — something you know
CBecause email-based codes expire too quickly to count as a second factor
DBecause VPN authentication does not support multifactor methods
✓ Correct! True MFA requires credentials from different factor categories. A password (knowledge) plus an email code (also knowledge — the code is in an inbox accessible via credentials you know) is two-step verification, not MFA. Additionally, email-based codes are not genuine ownership factors because the email account itself may be accessed from any device.
QUESTION 2 OF 3 · BIOMETRICS
A hospital deploys fingerprint scanners for physical access control. After configuration, staff are occasionally locked out despite presenting valid fingers, but no unauthorized individuals have ever gained access. Which metric best describes the problem, and what is the risk of fixing it by adjusting the tolerance threshold in the wrong direction?
AHigh FAR — reducing tolerance would further increase false lockouts
BHigh CER — this always increases when FAR decreases
CHigh FRR — loosening tolerance to fix it risks increasing FAR, potentially allowing unauthorized access
DHigh FER — staff could not enroll during the initial deployment
✓ Correct! Legitimate users being locked out = high FRR (False Rejection Rate / Type I error). To fix it, you loosen the tolerance threshold — but this risks raising the FAR (False Acceptance Rate / Type II error), which is the more dangerous failure. QUESTION 3 OF 3 · PASSWORD POLICY
A new CISO reviews the organization's password policy and finds it requires passwords to be changed every 45 days, must contain at least one uppercase letter, number, and special character, and shows users a hint on the login page. Which elements would current NIST guidance recommend removing?
AThe minimum length requirement only
BThe periodic rotation policy, the complexity rules, and the password hints

✓ Correct! Current NIST guidance deprecates all three: (1) mandatory periodic rotation — unless there is evidence of compromise, forced changes produce weaker passwords; (2) complexity rules — they drive predictable patterns like Password1!; (3) password hints — they reveal enough to assist attackers. NIST now favors length, uniqueness, and password managers.
4B

Topic 4B — Access Management

Authentication answers "who are you?" — Access Management answers "what are you allowed to do?" Once a subject is authenticated, a set of access control rules determines which resources they can reach, which actions they can perform, and what is explicitly off-limits.

4B Access Control Models

Discretionary & Mandatory Access Control

An access control model defines who has the authority to make access decisions and what criteria those decisions are based on. DAC and MAC represent the two ends of the spectrum — individual owner discretion versus system-enforced rules.

DAC

Discretionary Access Control

Every resource has an owner. The owner modifies the resource's Access Control List (ACL) to grant or revoke access at their discretion. Default model in Windows NTFS and most UNIX/Linux filesystems.

✓ Most flexible — no central admin required for every change

⚠ Weakest security — relies on every owner making correct decisions; highly vulnerable to insider threats and compromised accounts

Owner decidesACL-basedCommercial
MAC

Mandatory Access Control

Access decisions are removed from users entirely. Every object gets a classification label (Confidential / Secret / Top Secret). Every subject gets a clearance level. The rule: read at your level or below; write restrictions prevent downgrading classified content.

✓ Strongest control — non-discretionary, no user can override rules

⚠ Rigid and complex to administer; primarily used in government/military/intelligence

System decidesLabels + ClearanceGovernment/Military
MAC uses the rule "read down, write up" — a Top Secret user can READ Secret and Confidential data (down from their level) but cannot WRITE to lower-classified documents (which would downgrade the information). Most MAC implementations also add compartments — a file classified Secret/HR requires both Secret clearance AND HR compartment membership to access.
4B RBAC & ABAC

Role-Based & Attribute-Based Access Control

RBAC and ABAC sit between the extremes of DAC and MAC — they are non-discretionary like MAC, but far more flexible. These are the models most commonly deployed in real enterprise environments.

RBAC

Role-Based Access Control

Permissions are assigned to roles (e.g., Read Only, Administrator, Help Desk), not directly to individual users. Users inherit permissions by belonging to a role. Rights flow through role membership — never assigned directly. Administrators implement this using security groups.

✓ Scalable — change a group's permission once, all members update instantly

⚠ Cannot enforce contextual conditions (time, device, location)

Role = permission setSecurity groupsNon-discretionary
ABAC

Attribute-Based Access Control

Access decisions are based on a combination of subject attributes (who), object attributes (what), and context attributes (device OS, IP, time of day, patch status). The most fine-grained model available — can enforce policies RBAC cannot.

✓ Contextual — same user, same role, different conditions = different access decision

⚠ More complex to configure and manage than RBAC

Multi-attributeContext-awareMost granular
ABAC example — a Finance role user can access financial reports only if connecting from a corporate device, within the corporate network, during business hours. The same user on a personal unpatched device at 2 AM is denied — even though their role has not changed. ABAC can also implement M-of-N control (requiring multiple authorized people to jointly complete a task), separation of duties, and behavioral monitoring.
4B Rule-Based & Conditional Access

Rule-Based & Conditional Access Control

Rule-based access control is an umbrella term — any model where access policies are system-enforced rather than user-determined. MAC, RBAC, and ABAC all qualify. Conditional access extends this by monitoring behavior continuously throughout a session — not just at login.

MechanismPlatformHow It Works
UACWindowsPrompts for confirmation or re-authentication when an action requires elevated privileges. Prevents persistent elevation — each privileged action must be explicitly confirmed.
sudoLinux / UnixStandard user invokes elevated privileges for specific commands only. Elevation is task-specific and time-limited — not a permanent state.
Conditional Access PoliciesCloud IAM platformsMonitors location, device compliance, and behavior during a session. Can suspend an account or require re-authentication mid-session if conditions are triggered.
⚙️ CONDITIONAL ACCESS TRIGGERS
  • Location-based — unexpected geographic origin triggers re-authentication or access restriction
  • Device compliance — unpatched or non-compliant devices are denied or limited access even with valid credentials
  • Behavioral anomalies — unusual volume or pattern of access requests triggers a challenge
  • Continual authentication — authentication is not a one-time login event; it is an ongoing evaluation throughout the session
4B Least Privilege

Principle of Least Privilege

The Principle of Least Privilege states that every user, service account, and process should have only the minimum permissions required to perform their authorized tasks — and nothing more. Not approximately. Not a little extra. Precisely the minimum sufficient amount.

📌 WHY LEAST PRIVILEGE MATTERS
  • If a compromised account has minimal permissions, the damage a threat actor can cause is inherently limited
  • Least privilege begins at the design phase — business workflows must be analyzed to determine what roles and permissions each function actually requires before any access is provisioned
  • Authorization creep — even correctly provisioned accounts drift over time as users change roles, accumulate group memberships, or receive temporary elevations that are never revoked
  • Auditing is the continuous defense — regularly reviewing group memberships, ACLs, and temporary elevations is not optional; it is a core operational control
Authorization creep is silent. A user who has accumulated excessive permissions over years of role changes presents a significant risk — and it is often invisible until an incident occurs. Auditing must be scheduled and systematic, not reactive.
Misconfiguration TypeResultRisk
Too RestrictiveUsers cannot complete legitimate tasksSupport burden, workarounds, shadow IT
Too PermissiveUsers have more access than their role requiresLarger attack surface, data breach impact
4B Account Provisioning

User Account Provisioning & Deprovisioning

Access control models only work if accounts are set up correctly from the start and removed cleanly at the end. Think of user accounts as assets — like hardware or software — they require a defined lifecycle. Provisioning creates the account. Deprovisioning removes it.

1

Identity Proofing

Verify the person's identity using official documents before any account is created. Background checks may also apply depending on the role. If this step fails, everything built on it is compromised.

2

Issue Credentials

User selects a private password and/or enrolls with biometric or token-based authenticators — directly implementing the authentication technologies from Topic 4A.

3

Asset Allocation

Issue hardware and software needed for the role. If insufficient resources are provided, employees may source hardware/software independently — creating shadow IT: unmanaged devices outside security controls.

4

Security Education

Train the user on organizational policies, acceptable use, and security risks. A user who does not understand the policies cannot follow them — and ignorance does not reduce the risk they represent.

5

Permission Assignments

Configure access rights using the appropriate model (RBAC, MAC, ABAC). Tag privileged accounts for enhanced monitoring — elevated privileges carry elevated risk from day one.

Deprovisioning failures are among the most common sources of unauthorized access. Former employees whose accounts remain active, and stale contractor accounts from completed projects, are direct security risks. Remove from all roles and groups — disable before deleting to allow for audit requirements if applicable.
4B Account Attributes & Policies

Account Attributes & Access Policies

Every user account is a structured object with defined attributes and associated policies — not just a username and password. These attributes form the identity record the system uses for every access decision.

SECURITY IDENTIFIER (SID)

A unique, system-assigned value that identifies this account independent of name changes. The SID — not the account name — is what the OS references in access control decisions.

ACCOUNT NAME

The human-readable identifier used to log in and reference the account. Can change — the SID does not.

CREDENTIAL

The authentication mechanism — password hash, digital certificate, biometric enrollment, or hardware token association — linked to this account.

PROFILE

Stores extended attributes (name, email, department, manager), a home folder for personal data, and per-application settings that follow the user across devices.

⚙️ GROUP POLICY OBJECTS (GPOs) — WINDOWS AD
  • GPOs configure and deploy access rights centrally across Windows Active Directory environments
  • Can be linked to sites, domains, and Organizational Units (OUs) — applying policies broadly or narrowly
  • One GPO at the domain level can control rights for thousands of accounts simultaneously
  • Navigation path: Computer Configuration → Windows Settings → Security Settings → Local Policies → User Rights Assignment
4B Account Restrictions

Account Restrictions

Account restrictions operate as a second line of defense — even if valid credentials are stolen, policy-based controls can detect and block unauthorized use based on where and when the login is occurring.

LOCATION-BASED

Geographic & Network Restrictions

Restrict accounts based on IP address, subnet, VLAN, or OU. IP geolocation maps an IP to an approximate geographic location (country/city level, via ISP registry data). Location services provide more precise positioning using GPS, cell towers, Wi-Fi, and Bluetooth triangulation.

TIME-BASED

Time & Duration Restrictions

Time-of-day — sets authorized login hours (e.g., 7am–7pm weekdays only). Login attempts outside these hours are blocked regardless of credential validity. Duration-based — maximum continuous logged-in time to limit exposure from unattended sessions.

IMPOSSIBLE TRAVEL

Risky Login Detection

Tracks geographic locations of login events over time. If two logins occur from locations physically impossible to reach within the elapsed time — the account is automatically disabled and an alert is raised. Highly effective at detecting stolen credentials used remotely.

TEMP PERMISSIONS

Automatic Expiry Policy

Removes an account from a security role or group after a defined period — automatically, by policy. Directly addresses authorization creep by making temporary elevations self-revoking rather than dependent on administrator action.

4B Privileged Access Management

Privileged Access Management — PAM

Standard accounts have limited scope. Privileged accounts can install software, disable security controls, manage servers and databases, and modify network configurations. A compromised privileged account is a network-wide threat. PAM is the framework of controls specifically designed to address this.

🏗️ PAM CORE PRINCIPLES
  • Identify and document all privileged accounts — full visibility into what they are, who uses them, and how
  • Enforce strong credentials on privileged accounts — MFA or passwordless at minimum
  • Use Secure Administrative Workstations (SAWs) — dedicated devices with minimal attack surface for admin tasks only
  • Minimize the number of privileged accounts — more accounts = more attack surface
  • Never share accounts or use default accounts — shared accounts eliminate individual accountability
TEMPORARY ELEVATION

Account gains admin rights for a limited task period, then returns to standard access. UAC (Windows) and sudo (Linux) are the most common examples. Simplest ZSP model but least audit oversight.

PASSWORD VAULTING

Privileged accounts are checked out from a secure vault for a defined time. Access requires a logged justification. Approval can be automated (policy-based) or manual (M-of-N control). Full audit trail — every access is recorded.

EPHEMERAL CREDENTIALS

System generates a new credential for each specific task and destroys it immediately upon completion. Nothing persists for an attacker to steal. Closest equivalent to Perfect Forward Secrecy (Lesson 3) — applied to identity management.

Zero Standing Privileges (ZSP) — the gold standard for PAM. Elevated rights are not permanently assigned. Administrators must explicitly request the privileges they need, receive them for a limited period, and have them automatically revoked when the period expires. Just-in-Time (JIT) permissions is the mechanism that implements ZSP. Also note: PAM applies equally to service accounts — these are frequently overlooked but often carry significant system-wide access.

QUESTION 1 OF 3 · ACCESS CONTROL MODELS
A defense contractor requires that personnel with "Secret" clearance can never read documents labeled "Top Secret" — even if those documents are stored in a shared folder the user has mapped. Which access control model enforces this restriction, and what is the defining characteristic that makes it effective?
ARBAC — because roles are assigned by administrators, not individual users
BDAC — because the folder owner can configure the ACL to block Secret clearance accounts
CMAC — because access rules are system-enforced based on clearance and classification labels, and no individual user can override them
DABAC — because it uses subject attributes including clearance level to make access decisions
✓ Correct! MAC uses classification labels on objects and clearance levels on subjects to enforce access decisions at the system level. The defining characteristic is that these rules are non-discretionary — no user, regardless of seniority, can override them. DAC would allow the folder owner to bypass this by modifying the ACL. ABAC could approximate it but MAC is the model specifically designed for this use case.
QUESTION 2 OF 3 · LEAST PRIVILEGE
A security audit reveals that a marketing associate has Read/Write access to the financial reporting server, administrator rights on three application servers, and is a member of the VPN access group — all granted during various projects over the past three years. The user's current role requires none of these. What is the name of this condition and what is the primary remediation?
APrivilege escalation — the user actively requested and obtained elevated rights
BShadow IT — the user installed unauthorized applications to gain those permissions
CAuthorization creep — the remediation is a systematic privilege audit and immediate revocation of all excess rights
DDAC failure — resource owners failed to update ACLs when the user changed roles
✓ Correct! Authorization creep is the gradual accumulation of permissions that exceeds what a user's current role requires — typically through role changes, project-based temporary grants, and incremental group memberships that are never revoked. It is dangerous because it is silent. The remediation is a systematic audit — reviewing group memberships, ACLs, and temporary grants — and revoking anything not required by the current role.
QUESTION 3 OF 3 · PAM
An organization wants to implement the most rigorous privileged access control available. They want zero persistent credentials — each time an administrator needs to perform a task, a unique credential is created, used, and then permanently destroyed. Which PAM model describes this approach?
APassword vaulting — credentials are checked out from a vault for a defined period
BTemporary elevation — admin rights are granted for a limited task and automatically expire
CEphemeral credentials — a unique credential is generated per task and destroyed upon completion, leaving nothing persistent to steal
DSecure Administrative Workstations — dedicated devices with minimal attack surface for admin tasks
✓ Correct! Ephemeral credentials represent the gold standard — also called Zero Standing Privileges (ZSP) implemented via Just-in-Time (JIT) permissions. Because the credential exists only for the duration of a single task, there is nothing persistent for an attacker to steal or reuse — analogous to how Perfect Forward Secrecy in TLS uses ephemeral session keys that are discarded after each session.
4C

Topic 4C — Identity Management

Authentication and access management work well within a single organizational boundary. But modern organizations span on-premises networks, cloud services, and external partners. Federated identity management extends consistent identity governance across all of these environments — without requiring users to maintain separate accounts everywhere.

4C Authentication Providers

Local, Network & Remote Authentication

Every operating system has an authentication provider — the software architecture that handles credential verification before granting a shell session. A foundational principle applies to all: plaintext passwords are never stored or transmitted. Only cryptographic hashes are compared.

ScenarioWindows MechanismLinux Mechanism
Local Sign-In LSASS compares submitted hash against SAM database (part of the Windows registry). Also called interactive logon. Password hash stored in /etc/shadow. Account names in /etc/passwd.
Network Sign-In LSASS passes credentials to an Active Directory domain controller. Preferred protocol: Kerberos. Legacy fallback: NTLM. SSH for interactive network login — can authenticate with cryptographic keys instead of passwords.
Remote Sign-In Authentication over VPN, enterprise Wi-Fi, or web portal — establishes a secure channel between client, remote access device, and authentication server. PAM (Pluggable Authentication Modules) — flexible framework for enabling different providers (smart cards, network directory services).
4C Directory Services

Directory Services & LDAP

A directory service is a centralized database of subjects — users, computers, security groups, and services — along with their attributes and Access Control Lists. It is the authoritative source for both identity information and authorization decisions across the enterprise.

X.500

The original directory standard. The foundation from which LDAP was developed. Still referenced as the architectural basis of modern enterprise directories.

LDAP

Lightweight Directory Access Protocol — the common query language that allows different vendor products to interoperate. A Linux client, Windows server, and network appliance can all query the same LDAP directory.

SCHEMA

Defines what types of attributes exist in the directory, what information they can contain, and whether they are required or optional for each object type.

ACL STORAGE

The directory also stores Access Control Lists — making it the single authoritative source for both identity and authorization data.

Every object in an LDAP directory is identified by a Distinguished Name (DN) — a sequence of attribute-value pairs, listed from most specific (left) to broadest (right). The leftmost attribute alone is the Relative Distinguished Name (RDN).

4C Kerberos & SSO

Single Sign-On — Kerberos Authentication

SSO (Single Sign-On) lets users authenticate once and access multiple integrated services without re-entering credentials. Kerberos is the protocol that implements SSO in most enterprise Windows environments (Active Directory) — named after the three-headed mythological guardian.

KEY DISTRIBUTION CENTER (KDC)

The trusted intermediary that vouches for both clients and application servers. Contains two services: the Authentication Service (AS) and the Ticket Granting Service (TGS).

AUTHENTICATION SERVICE (AS)

Handles initial login. Verifies the user's credentials and issues a Ticket Granting Ticket (TGT) — a logical token proving the user has been authenticated.

TICKET GRANTING SERVICE (TGS)

Uses the TGT to issue service tickets for specific application servers. The TGT alone does not grant resource access — the TGS must issue a service-specific ticket.

PRINCIPAL

The user or service requesting access. Presents encrypted timestamps (using their password hash as the key) to prove credential knowledge — the password hash itself never travels the network.

1
Principal encrypts the current timestamp using their password hash and sends it to the Authentication Service as a logon request. The password hash itself is never transmitted.
2
The AS verifies: account exists in AD, the hash decrypts the timestamp correctly, and the timestamp has not expired (preventing replay attacks).
3
If valid, the AS returns: (a) a TGT encrypted with the KDC's secret key (the principal cannot read or modify it), and (b) a TGS session key encrypted with the user's password hash.
The TGT is a logical token — it confirms the user has been authenticated. It does NOT by itself grant access to any specific resource. Access to individual services requires the second phase: the Ticket Granting Service. Timestamps throughout the process prevent replay attacks.
4C Kerberos TGS Flow

Kerberos Ticket Granting Service Flow

Phase 2 of Kerberos — the TGS flow — converts the TGT (proof of authentication) into a service ticket that grants access to a specific resource. This is where authentication transitions into authorization.

1
Service Request — Principal sends the TGS: a copy of the TGT, the target application server name, and a time-stamped authenticator encrypted with the TGS session key. TGS verifies both and confirms the ticket has not expired or been replayed.
2
Service Ticket Issued — TGS returns: (a) a service session key encrypted with the TGS session key, and (b) a service ticket containing the user's identity, SID, group SIDs, and service session key — encrypted with the application server's secret key. Only the application server can decrypt it.
3
Presents Service Ticket — Principal forwards the service ticket + a new time-stamped authenticator (encrypted with service session key) to the application server. The server decrypts both and confirms the message is untampered.
4
Mutual Authentication (optional) — The server encrypts the timestamp back using the service session key. The principal decrypts and verifies it — confirming the server is also trustworthy. Prevents on-path (MITM) attacks where an attacker impersonates the server.
5
Data Transfer — Authentication is complete. The server responds to access requests subject to its own ACL.
The KDC is a single point of failure — if it goes down, no authentication can occur across the domain. In practice, Active Directory mitigates this by running the KDC service on multiple domain controllers simultaneously, providing redundancy and resilience.
4C Federation

Federated Identity Management

Federation means that one organization trusts identity accounts created and managed by a different network. Rather than each organization maintaining separate accounts for every partner or customer, a trust relationship allows identities to be accepted across boundaries.

Consumer example: signing in to a third-party service using your Google or Microsoft account. Google acts as the Identity Provider; the third-party site is the Service Provider. A trust relationship was established between them — you authenticate once, and that authentication is honored across the boundary.
SERVICE PROVIDER (SP)

The application or resource the user wants to access. Does not authenticate users directly — redirects to the Identity Provider and trusts the result.

IDENTITY PROVIDER (IdP)

The organization that manages and vouches for the user's identity. Authenticates the user using its own methods and issues a signed claims token as proof.

CLAIMS TOKEN

A signed document asserting the user's identity and attributes. Signed with the IdP's private key so the SP can verify it came from a trusted source — without the user creating a separate SP account.

PRE-ESTABLISHED TRUST

Before federation can work, the SP and IdP must agree on the trust relationship in advance. Without this prior agreement, the SP has no basis for accepting the IdP's claims.

1

Pre-established trust between SP and IdP

Foundation of the entire federation model — without this, step 3 cannot occur.

2

User initiates session with SP

Attempts to access the service provider resource.

3

SP redirects to IdP for authentication

SP does not verify the user — it trusts the IdP to do so.

4

User authenticates to IdP

Using the IdP's own methods — password, MFA, SSO — whatever the IdP supports.

5

IdP issues signed claims token

Token is signed with the IdP's private key — the SP can verify the signature without ever seeing the user's credentials.

6

SP validates token and grants access

SP verifies the signature, associates the principal with an account and permissions in its own system. No separate account creation required.

4C OAuth & JWT

Open Authorization — OAuth & JSON Web Tokens

Enterprise federation protocols like SAML are powerful but heavyweight. Modern web and mobile apps need something lighter. OAuth (Open Authorization) is the framework designed for RESTful APIs — it communicates authorizations between services without exposing user passwords to consumer applications.

OAuth is primarily an authorization framework, not a pure authentication protocol. It enables resource sharing between sites without the user ever giving their password to the consumer app — the Identity Provider handles authentication; the consumer site receives only an access token.
RESOURCE OWNER

The user who owns the account and data being requested.

OAUTH CLIENT

The application requesting access to part of the user's account data. Must register with the authorization server to receive a Client ID and Client Secret.

AUTHORIZATION SERVER

Processes authorization requests and issues access tokens after the user approves. A single authorization server can manage multiple resource servers.

RESOURCE SERVER

Hosts the actual data and functions the OAuth client wants to access (also called the API server). Accepts access tokens and fulfills requests when they are valid.

🔑 JSON WEB TOKENS (JWT)
  • OAuth uses JWT format for claims data — a compact, URL-safe encoding of identity and authorization information
  • JWTs can be passed as Base64-encoded strings in URLs and HTTP headers — lightweight and compatible with RESTful APIs
  • JWTs can be digitally signed — providing both authentication of the source and integrity of the claims data they carry
  • The client receives an access token from the authorization server, presents it to the resource server, and the resource server fulfills the request if the token is valid

Topic 4C — Knowledge Check

QUESTION 1 OF 3 · KERBEROS
During a Kerberos authentication, a user's workstation sends an encrypted request to the Authentication Service. A network analyst captures this traffic and attempts a replay attack — resending the same captured packet 10 minutes later. Which Kerberos design element prevents this attack from succeeding?
AThe TGT is encrypted with the KDC's secret key and cannot be decrypted by the attacker
BThe timestamp embedded in the request — Kerberos rejects requests where the timestamp exceeds the allowed time window, making replays invalid
CThe password hash is never transmitted, so the replayed packet contains no usable credential
DThe domain controller checks the Active Directory database for duplicate requests
✓ Correct! Kerberos embeds a timestamp in every encrypted request. The Authentication Service checks that the timestamp falls within an allowed window (typically 5 minutes). A replayed packet from 10 minutes earlier falls outside this window and is rejected. This is why Kerberos requires synchronized clocks across domain clients and servers.
QUESTION 2 OF 3 · LDAP
A network administrator needs to reference a specific database server in the directory. The server is named PRODDB01, located in the IT Operations department, owned by WidgetCorp, located in the US, in the domain corp.widget.com. Which of the following Distinguished Names correctly identifies this object?
ADC=corp, DC=widget, DC=com, O=WidgetCorp, OU=IT Operations, CN=PRODDB01
BCN=PRODDB01, OU=IT Operations, O=WidgetCorp, C=US, DC=corp, DC=widget, DC=com
COU=IT Operations, CN=PRODDB01, C=US, DC=corp, DC=widget, DC=com
DCN=PRODDB01, DC=corp, DC=widget, DC=com, OU=IT Operations, O=WidgetCorp
✓ Correct! LDAP Distinguished Names are written from most specific to broadest, left to right. The object itself (CN) comes first, then the container (OU), then the organization (O), country (C), and finally the domain components (DC). Option A reverses the order. Options C and D place attributes out of sequence.
QUESTION 3 OF 3 · FEDERATION / OAUTH
A developer builds a fitness app that requests access to a user's Google Calendar to schedule workout reminders. The user is redirected to Google to approve the request, then returned to the fitness app with an access token. The fitness app never sees the user's Google password. Which framework is in use, and what role does Google play?
AKerberos — Google is acting as the Key Distribution Center
BSAML — Google is acting as the Service Provider
COAuth — Google is acting as both the Identity Provider / Authorization Server and the Resource Server; the fitness app is the OAuth client
DLDAP — Google is the directory service authenticating the user and returning group membership attributes
✓ Correct! This is the OAuth framework. The fitness app (OAuth client) requests access to Google Calendar data (Resource Server). The user approves through Google (Authorization Server / IdP), which issues an access token. The fitness app presents this token to access the calendar — never receiving the user's Google password. This is the core value of OAuth: resource sharing without credential exposure.

Lesson 4 · Complete Summary

What You Have Learned

🔑

IAM Framework

Identity and Access Management is a discipline — technologies, policies, and processes ensuring the right people access the right resources. Built on two pillars: authentication and authorization.

🧠

Authentication Design

Supplicant presents credentials to authentication server. Must satisfy CIA: confidentiality (no plaintext), integrity (no forgery), availability (no lockouts). Knowledge factor: passwords, passphrases, PINs.

📋

Password Policy

Length is the strongest control. NIST deprecates complexity rules, mandatory rotation, and hints. Prioritizes length + uniqueness + password managers. Credential stuffing targets password reuse.

🔒

MFA & Factor Types

True MFA requires different factor categories. Know: Something you know, have, are, somewhere you are. 2FA = two factors specifically. SMS/email = two-step verification, not true MFA.

👁️

Biometrics

FRR = false rejections (Type I). FAR = false acceptances (Type II) — more dangerous. CER = where FRR equals FAR — lower is better. Facial recognition uses infrared to defeat 2D spoofing.

🪙

Hard Tokens

Certificate-based (PKI required), OTP/TOTP/HOTP (shared secret risk), FIDO U2F (no shared secret). Static tokens = vulnerable to replay. Dynamic tokens defeat replay attacks.

📱

Soft Tokens & Passwordless

Authenticator apps = true ownership factor (locally generated). SMS/email = two-step verification only. FIDO2/WebAuthn = fully passwordless, private key never leaves device, no shared secret on network.

🏛️

Access Control Models

DAC: owner decides (flexible, weakest). MAC: system decides by labels (rigid, strongest). RBAC: roles carry permissions (scalable). ABAC: attributes + context (most granular). Rule-based = any system-enforced model.

🎯

Least Privilege

Minimum necessary permissions. Starts at design phase. Authorization creep is silent — systematic auditing is the defense. Both too-restrictive and too-permissive configurations carry costs.

👤

Provisioning & PAM

Identity proofing → credentials → assets → training → permissions. Deprovisioning failures = unauthorized access. PAM: SAWs, ZSP, JIT permissions. Ephemeral credentials = most rigorous model.

🌐

Kerberos & SSO

Authenticate once, access many. AS issues TGT (proves identity). TGS issues service ticket (grants access). Timestamps prevent replay. Mutual auth prevents on-path attacks. KDC = single point of failure.

🔗

Federation & OAuth

Federation = cross-org identity trust via claims tokens. IdP authenticates; SP accepts the signed token. OAuth = authorization framework for RESTful APIs. JWT = claims format. User password never leaves the IdP.
