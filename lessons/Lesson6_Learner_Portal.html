6A

Topic 6A — Cloud Infrastructure

Before you can secure the cloud, you need to understand how it is organized, owned, and operated. This topic covers the full cloud infrastructure stack — from deployment models through security controls.

6ACloud Platform Concepts

Cloud Platform Concepts

Cloud network architecture encompasses concepts and technologies designed to ensure the confidentiality, integrity, and availability of data and applications within cloud-based environments. The CIA Triad still applies — the environment changes, the principles do not.

⚡
ON-DEMAND PROVISIONING

Resources deployed immediately — no waiting for physical hardware. Computing power, storage, and network on request.

📈
ELASTICITY

Resources expand when demand rises and contract when demand drops — automatically and dynamically.

📊
SCALABILITY

The environment grows to handle larger workloads over time as organizational needs increase.

🏘️
MULTI-TENANCY

Multiple organizations share the same physical infrastructure with logical separation. Isolation is critical.

💻
VIRTUALIZATION

One physical machine runs multiple isolated virtual environments simultaneously.

⚖️
LOAD BALANCING

Distributes traffic evenly across servers — maintains performance, supports availability, provides DDoS protection.

🔄
AUTO-SCALING

Resources adjust in real time based on workload. New instances must inherit security configurations automatically.

🌐
HYBRID / MULTI-CLOUD

Reduces vendor lock-in and improves resilience by spreading workloads across providers.

Every cloud feature has a security implication. Auto-scaling that launches instances without security controls creates immediate vulnerabilities. Speed and misconfiguration happen at the same pace.
6ADeployment Models

Cloud Deployment Models

A cloud deployment model classifies how a cloud service is owned and provisioned. Different models carry different threats and vulnerabilities — knowing the model helps you understand the risk.

PUBLIC (MULTI-TENANT)

Offered over the internet by a CSP to the general public. Multiple customers share the same infrastructure. Cost-effective — but shared resources introduce risk.

PRIVATE

Infrastructure completely owned by and dedicated to one organization. No sharing. Preferred by banking and government. On-site or off-site. Highest control and cost.

HOSTED PRIVATE

Hosted by a third party for the exclusive use of one organization. More secure than public — better guaranteed performance. More expensive.

COMMUNITY

Several organizations share costs of a private cloud. They share a common concern — regulatory compliance or shared security standards. Example: multiple hospitals sharing HIPAA-compliant infrastructure.

HYBRID CLOUD

Combines public and private cloud. Sensitive data in the private cloud — less sensitive workloads run publicly. Greater flexibility — more complex to secure.

MULTI-CLOUD

Uses services from multiple public CSPs simultaneously. Reduces vendor lock-in — adds monitoring complexity. More environments to secure.

Travel Company Example: A travel company runs its sales website on a private cloud most of the year. During peak travel seasons, it expands to a public cloud to handle traffic spikes, then scales back. This is hybrid cloud in action.
Hybrid cloud security challenges: managing multiple environments, enforcing consistent policies, on-premises vs. cloud boundary confusion, data consistency issues, and latency. Know SLA and vendor lock-in as key exam terms.
6AService Models

Cloud Service Models — XaaS

Cloud services are differentiated by their level of complexity and pre-configuration. These models are grouped under XaaS — "anything as a service." Your security responsibility shifts with each model.

SaaS
Software as a Service — Provider hosts fully built, ready-to-use applications. No installation. Examples: Microsoft 365, Google Workspace, Salesforce. Provider manages everything — you manage access and data.
PaaS
Platform as a Service — Provider delivers a development platform. Your team builds applications on top. Examples: Azure SQL Database, Google App Engine. You secure what you build — the provider secures the platform.
IaaS
Infrastructure as a Service — Provider delivers raw infrastructure — servers, load balancers, storage. You rent it and build everything on top. Examples: Amazon EC2, Azure VMs. You secure everything above the hardware layer.
FaaS
Function as a Service — Serverless computing. Individual functions execute in response to triggers (HTTP requests, database events). Provider manages all infrastructure. You secure access and data.
⚠ Third-Party Vendor Considerations
  • SLA — defines uptime, performance, and support commitments. Includes penalties for non-compliance.
  • Vendor lock-in — over-dependence on one provider. Mitigate with multi-cloud strategies and data portability planning.
  • ISA (Interconnection Security Agreement) — defines encryption, access controls, audit rights, and compliance obligations.
  • Always assess vendor security: encryption, patch management, incident response, and regulatory compliance (GDPR, HIPAA, PCI-DSS).
Security responsibility by model — SaaS: you secure access and data. PaaS: you secure the application you build. IaaS: you secure everything above the physical hardware. Data classification is ALWAYS the customer's responsibility.
6AResponsibility Matrix

The Shared Responsibility Model

When you move to the cloud, security risks are not transferred to the provider — they are shared. Understanding exactly where the boundary lies is one of the most tested concepts on the Security+ exam.

☁️ Cloud Service Provider Owns

  • Physical security of infrastructure and data centers
  • Securing compute, storage, and network equipment
  • Foundational networking and DDoS protection
  • Cloud storage backup and recovery
  • Resource isolation among tenants
  • Infrastructure-level identity and access control
  • Security monitoring for infrastructure
  • Managing and securing data centers globally

🏢 Customer Organization Owns

  • User identity management
  • Geographic configuration for data and services
  • User and service access controls
  • Data and application security configuration
  • OS protection when deployed
  • Encryption configuration and key management
  • Data classification — ALWAYS the customer
A misconfigured cloud storage bucket that exposes customer data is entirely the customer's failure — not the provider's. The provider made the bucket available; the customer left it publicly accessible. Misconfiguration — not hacking — is the leading cause of cloud breaches.
Physical security is ALWAYS the provider's. Data classification and accountability are ALWAYS the customer's. The gap between those two is where vulnerabilities live.
6AArchitecture Models

Centralized vs. Decentralized Computing

CENTRALIZED

All processing in one location. All users depend on that server. Examples: mainframes, client-server. Easy to control — single point of failure.

DECENTRALIZED

Processing distributed across multiple locations. No single point of failure or control. More resilient — harder to monitor consistently.

BLOCKCHAIN

Distributed ledger. Transactions recorded in linked, tamper-resistant blocks copied across many nodes. Used in cryptocurrency, supply chain, and identity verification.

PEER-TO-PEER (P2P)

No central server — every device acts as client and server. Resilient but hard to monitor. Can distribute malware.

CDN

Content Delivery Network — distributes content across geographically dispersed servers. Reduces latency, improves performance, provides built-in DDoS protection. Examples: Cloudflare, Akamai.

TOR

The Onion Router — anonymization network routing traffic through volunteer relay nodes. Legitimate uses: privacy protection for journalists. Security concern: used by threat actors to hide malicious traffic.

TOR exit node traffic in network logs is a potential indicator of suspicious activity. Know both legitimate uses and malicious applications of TOR for the exam.
6AVirtualization

Application Virtualization & Containerization

Virtualization allows multiple workloads to run on the same physical hardware — each isolated from the others. The distinction between VMs and containers is heavily tested on Security+.

App Virtualization
Only the application is virtualized — not a full desktop. Client accesses an app hosted on a server or streamed via HTML5. Products: Citrix XenApp, Microsoft App-V, VMware ThinApp.
Container
A portable, self-contained unit with application code, libraries, and configuration. Behaves consistently on any platform. Managed by Docker. Solves the dependency nightmare — each container holds its own independent copy of dependencies.
Type 1 Hypervisor
Bare-metal — runs directly on physical hardware. No host OS underneath. High performance. Enterprise environments. Examples: VMware ESXi, Microsoft Hyper-V.
Type 2 Hypervisor
Hosted — runs on top of a host OS. Used for dev, testing, and personal labs. Examples: VMware Workstation, Oracle VirtualBox.
DimensionVirtual MachineContainer
OSEach VM has its own complete OSShares the host OS kernel
IsolationStronger — own OS layerLighter — share kernel
PerformanceMore resource-intensiveLighter and faster
Security RiskVM escape is difficultContainer escape is a real attack vector
Use CaseEnterprise production workloadsMicroservices, CI/CD, dev environments
Know Type 1 vs. Type 2 hypervisor product examples for the exam. VMs = stronger isolation, own OS. Containers = lighter, faster, share host OS kernel — smaller isolation boundary. Container escape is a tested attack vector.
6AServerless & Microservices

Serverless Computing & Microservices

VPC

Virtual Private Cloud — a private network segment within a public cloud. Logically isolated from other tenants. The organization controls IP ranges, subnets, routing, and firewalls.

SERVERLESS COMPUTING

Provider manages all infrastructure and scales automatically. Pay only for actual execution time. Examples: AWS Lambda, Google Cloud Functions, Azure Functions. You secure access and data.

MICROSERVICES

Applications built as a collection of small, independent services — each with a single responsibility. Teams work independently. Risk: integration issues increase attack surface between services.

EVENT-DRIVEN

Serverless functions triggered by specific events — HTTP requests, database changes, sensor readings. Each function handles one task and passes results forward. Replaces monolithic design.

🔑 Serverless Security Responsibilities
  • The provider manages and secures all physical infrastructure — no patching, no server administration for the customer
  • The customer remains responsible for securing access to functions and protecting data
  • The primary security job: ensuring clients accessing services have not been compromised
  • Microservices and IaC work together — IaC automates consistent, secure deployment of each microservice
6ACloud Automation

Infrastructure as Code & Cloud Automation

Infrastructure as Code (IaC) manages computing infrastructure using machine-readable definition files — making deployment automated, consistent, and reproducible.

IaC
Code that describes exactly what infrastructure you want. Written in YAML, JSON, or HCL (HashiCorp Configuration Language — used by Terraform and Consul). Version-controlled like application code.
Load Balancing
Distributes incoming traffic evenly across multiple servers. Prevents overload. Acts as a proxy. Algorithms consider capacity, response time, and workload. Provides built-in DDoS protection.
Edge Computing
Moves processing closer to users and devices to reduce latency. Critical for IoT, CDNs, and real-time applications. Security concern: edge nodes may have weaker physical security than central data centers.
Auto-Scaling
Automatically adds resources during high demand — releases them when demand drops. Must work with IaC to ensure new resources inherit security configurations immediately at launch.
IaC security risk: a misconfiguration in IaC code is replicated consistently across every environment it deploys. Code review of IaC files is a critical security practice. Auto-scaled resources without inherited security controls create immediate vulnerabilities.
6ASDN

Software-Defined Networking (SDN)

SDN separates network functions into three distinct planes — making centralized, automated policy enforcement possible at scale. The SDN controller is the brain and the highest-value attack target.

1

Management Plane — Visibility

Monitors traffic conditions and overall network status. Where administrators observe what is happening. If compromised, attackers can operate invisibly.

2

Control Plane — Decisions

Makes decisions about traffic prioritization, security, and routing. Where policy is defined. The SDN controller lives here. The highest-value attack target in any SDN environment.

3

Data Plane — Execution

Handles actual switching and routing of traffic. Enforces security access controls based on control plane instructions. Where traffic inspection and filtering physically occurs.

InterfaceDirectionWhat It Connects
Northbound APIUpward ↑SDN applications → SDN controller. Policy decisions communicated upward.
Southbound APIDownward ↓SDN controller → network appliances. Enforcement instructions pushed down to hardware.
NFVHorizontalNetwork Functions Virtualization — runs firewalls, routers, switches as software on VMs and containers.
SDN centralizes control — the controller is a single high-value target. Every Northbound and Southbound API interface is a potential attack surface. Know: Management (monitors) → Control (decides) → Data (executes).
6AArchitecture Features

Cloud Architecture Features & Considerations

CAPEX vs OPEX

CapEx — large up-front hardware investments. OpEx — cloud's pay-as-you-go model. Cloud converts CapEx to OpEx. Security must drive decisions — not cost alone.

SCALABILITY

Scale-up (vertical) — more CPU/memory on existing resource. Scale-out (horizontal) — add more instances. Both must automatically inherit security configurations.

FAULT TOLERANCE

Redundant hardware, fault tolerance clustering, data replication across multiple servers and data centers. Directly supports Availability in the CIA Triad.

HIGH AVAILABILITY

99.99% uptime guarantee. Three tiers: Local replication → Zone-Redundant → Geo-Redundant Storage (GRS). GRS is the highest tier — protects against regional disasters.

SLA vs ISA

SLA — performance and availability commitments with penalties. ISA (Interconnection Security Agreement) — encryption methods, access controls, audit rights, and compliance obligations.

PUE

Power Usage Effectiveness — measures data center energy efficiency. Lower PUE = more energy goes to computing. Redundant power: UPS systems, backup generators, multiple utility feeds.

SLA vs. ISA — SLA defines service performance and availability. ISA defines security responsibilities and compliance. GRS is the highest replication tier. Scale-up = vertical. Scale-out = horizontal. These distinctions are frequently tested.
6ACloud Security

Cloud Security Considerations

🔑 Data Protection in the Cloud
  • Data stored in the cloud is essentially stored on the internet — configuration mistakes have disastrous consequences
  • Apply strict access controls and encryption — both at rest and in transit
  • Manage encryption keys carefully — exposed keys render encryption meaningless
  • Organizations must still maintain their own disaster recovery plans — cloud resilience is not a substitute
⚙ Patching in the Cloud
  • Cloud patching is complex — large environments make it hard to identify every vulnerable component
  • Some providers do not allow customers to modify underlying infrastructure — creating compliance timeline challenges
  • Best practices: automated patch management, centralized control, regular update cycles, security monitoring
  • The ISA must address patching obligations when the provider controls the schedule
SD-WAN
Software-Defined Wide Area Network — connects branch offices, data centers, and cloud infrastructure via WAN with centralized policy management. Features: encryption, traffic segmentation, intelligent routing by application type, firewall integration. Secures the network path.
SASE
Secure Access Service Edge (pronounced "sassy") — combines WAN technologies with cloud-based security services. Operates under a Zero Trust model. Features: IAM, intrusion prevention, malware protection, content filtering. Secures the access layer for any user from any location.
SD-WAN secures the network path. SASE secures the access layer — applying identity-based, Zero Trust controls to every connection. SASE is cloud-delivered and built on Zero Trust from the ground up — not an add-on.

6B

Topic 6B — Embedded Systems & Zero Trust Architecture

As embedded systems and IoT proliferate, the traditional perimeter dissolves. Zero Trust Architecture provides the framework for an environment where nothing should be trusted by default — regardless of where it sits.

6BEmbedded Systems

Embedded Systems & Real-Time Operating Systems

An embedded system is a specialized computer built into a device to control specific functions. They are increasingly network-connected — making them critical security targets across every industry.

HOME APPLIANCES

Refrigerators, washing machines, coffee makers. Any connected device — however trivial — is a potential network entry point.

SMARTPHONES & TABLETS

Multiple embedded systems — processors, GPS, Wi-Fi, Bluetooth, NFC, cellular. High-value targets carrying credentials and corporate access.

AUTOMOTIVE SYSTEMS

Engine control units, entertainment, active safety systems. Connected vehicles create remote attack opportunities that directly affect physical safety.

INDUSTRIAL AUTOMATION

Robots, assembly line controllers, industrial sensors. Disruption can halt production or cause physical damage and worker injury.

MEDICAL DEVICES

Pacemakers, insulin pumps, glucose monitors. Life-critical functions. A compromised device can directly endanger a patient's life.

AEROSPACE & DEFENSE

Aircraft, satellites, military equipment — navigation, communication, and control. Consequences of compromise can be catastrophic.

🔑 RTOS — Real-Time Operating Systems
  • RTOS — purpose-specific OS for real-time processing with guaranteed response times. High stability and speed.
  • VxWorks — aerospace and defense (aircraft control, missile guidance)
  • FreeRTOS — open-source, robotics, industrial automation, consumer electronics
  • AUTOSAR — automotive framework (engine control, transmission, active safety)
  • Siemens SIMATIC WinCC — industrial automation and process control
  • Security breaches in RTOS environments can cause physical harm to people or equipment — these are national security concerns
RTOS security breaches are not just IT incidents — a compromised pacemaker, insulin pump, or industrial control system can injure or kill people. Know all four RTOS examples for the exam.
6BICS / SCADA

Industrial Control Systems & SCADA

ICS controls machinery used in critical infrastructure — power, water, healthcare, telecommunications, and national security. These environments reverse the standard CIA Triad priority.

PLC
Programmable Logic Controller — the core computing device in an ICS. Directly controls actuators (valves, motors, circuit breakers) and reads sensors (temperature, pressure). Connected via OT fieldbus or industrial Ethernet.
HMI
Human-Machine Interface — how operators interact with PLCs. Can be a physical control panel or software on a workstation. Common ICS attack target — compromising an HMI gives control over industrial processes.
DCS
Distributed Control System — manages process automation within a single site. A control server governs the entire automation system across multiple PLCs.
SCADA
Supervisory Control and Data Acquisition — large-scale, multi-site ICS. Runs on standard computers, manages field devices via WAN communications (cellular, satellite).
Data Historian
Specialized database recording all control loop information — sensor readings, process states, and operational events. A target for attackers seeking to understand system behavior before launching attacks.
A

Availability — #1 Priority

I

Integrity — #2 Priority

C

Confidentiality — #3 Priority

ICS/SCADA environments use the AIC Triad — Availability comes first. Hazardous electromechanical components are involved and safety is the overriding concern. A system that stops unexpectedly can cause immediate physical danger.
Stuxnet (2010): A worm targeting SCADA management software running on Windows PCs — specifically designed to attack centrifuges in Iran's nuclear fuel program. It manipulated PLC instructions to damage centrifuges while reporting normal operation to operators. The first cyberattack proven to cause real physical destruction. Reference: NIST SP 800-82 for ICS/SCADA security controls.
6BIoT

Internet of Things (IoT)

The IoT is the network of physical devices, vehicles, appliances, and objects embedded with sensors, software, and connectivity — enabling them to collect and exchange data. IoT introduces unique security challenges at massive scale.

SMART HOMES

IoT controls lighting, temperature, and security systems. Any connected device — however trivial — is a potential network entry point.

SMART CITIES

IoT manages traffic, monitors air quality, and improves public safety. Disruption affects entire communities simultaneously.

HEALTHCARE IOT

Wearables and implantable devices transmit patient data to providers. Compromised devices can directly endanger patients.

AGRICULTURE

Soil sensors, weather monitoring, crop tracking. Large deployments in remote areas make security monitoring complex.

⚠ IoT Security Risks
  • Designed for functionality — not security: limited processing power prevents strong security controls
  • Default credentials: many devices ship with simple, well-known passwords that users never change
  • Infrequent patching: many devices rarely or never receive firmware updates after release
  • Rushed to market: insufficient security testing before deployment
  • Lack of standardization: incompatible protocols make consistent security enforcement difficult
Mirai Botnet: Malware infected millions of IoT devices via default credentials — cameras, DVRs, routers — and weaponized them for massive DDoS attacks taking major websites offline for hours. The devices were not the target — they were the weapon.

Casino Fish Tank Hack: Attackers breached a casino through a smart thermometer in a fish tank connected to the corporate network. Network segmentation would have contained this breach immediately.
IoT security starts with: inventory all devices, change default credentials immediately, update firmware regularly, segment IoT devices onto separate network segments, monitor IoT traffic continuously. Know IoTSF, IIC, CSA, and ETSI as the major IoT security frameworks.
6BDeperimeterization

Deperimeterization — Why the Old Model Failed

The traditional security model assumed a clear boundary between trusted inside and untrusted outside. Five trends dissolved that perimeter completely — forcing a fundamental rethinking of security architecture.

CLOUD

Enterprise infrastructure spans on-premises and cloud platforms globally. Perimeter-based controls cannot protect resources that exist outside the traditional boundary.

REMOTE WORK

Employees connect from home networks and public Wi-Fi. Every home network becomes an extension of the corporate environment — and a potential entry point.

MOBILE / BYOD

Smartphones and tablets are primary computing devices. Personal devices often lack enterprise security controls. A compromised personal device directly threatens corporate data.

OUTSOURCING

Contractors and MSPs are given remote access to internal systems. A compromise at the contractor level can cascade directly into the organization's environment.

WIRELESS

Wireless networks are susceptible to many exploits. Public Wi-Fi is often open and unsecured. Network security keys are frequently well-known — defeating encryption entirely.

RESPONSE

Security shifts from defending boundaries to protecting individual resources and data. Controls travel with the asset regardless of location. Zero Trust is the framework built to implement this.

6BZero Trust

Zero Trust Architecture

Zero Trust assumes that all devices, users, and services are not inherently trusted — regardless of location. Core principle: Never trust, always verify. Every access request must be authenticated and continuously validated.

Adaptive Identity
Identity verification is continuous and context-based — adjusted by what the user is doing, what they are accessing, where they are, and what device they are using. Not a one-time login check.
Threat Scope Reduction
Access granted on a need-to-know basis only. Users and devices receive access only to specific resources required for a specific task. Reduces attack surface and limits damage from successful attacks.
Policy-Driven Access
Access determined by user identity + device posture + network context. Device posture = security configurations, software versions, and patch levels of the requesting device.
Policy Engine
Makes dynamic per-request authentication and authorization decisions using identity, credentials, threat intelligence, behavioral analytics, and host scanning results. The brain of the Zero Trust control plane.
Policy Administrator
Manages access tokens and establishes or tears down sessions based on Policy Engine decisions. Acts as the interface between the control plane and the data plane.
Policy Enforcement Point
Mediates every access request. The only process permitted to interface with the Policy Administrator. Sets up a secure data pathway if approved — tears down the session if denied or revoked.
Implicit Trust Zone
The encrypted, transient data pathway established between the PEP and the resource for an approved session. Made as small and short-lived as possible. Example: an IPsec tunnel.
Microsegmentation
Trust granted per transaction — not per session. Even a verified user can be blocked mid-session if behavioral analytics flag suspicious activity.
Google BeyondCorp: One of the most recognized Zero Trust implementations globally. Uses identity verification, device verification, and access control policies. Employees access company resources remotely without a VPN — proven at global scale.

NIST SP 800-207 is the foundational Zero Trust Architecture standard. CISA Zero Trust Maturity Model provides progressive implementation guidance.
Control plane: Policy Engine + Policy Administrator. Data plane: Policy Enforcement Point + Implicit Trust Zone. Know NIST SP 800-207 as the foundational ZTA reference document. Microsegmentation = trust per transaction, not per session.

Virtualization & Containers

Type 1 (bare-metal) vs. Type 2 (hosted). VMs = stronger isolation. Containers = lighter, share host OS kernel. Container escape is real.

⚡

Serverless & Microservices

Provider manages all infrastructure. Customer secures access and data. Integration issues between microservices are the primary risk.

💻

IaC & Automation

YAML, JSON, HCL (Terraform). Load balancing, edge computing, auto-scaling. Security must be built into automation from day one.

🔀

SDN

Management (monitors) → Control (decides) → Data (executes). Northbound API: apps to controller. Southbound API: controller to devices.

🛡️

Cloud Security

Data protection, patching complexity. SD-WAN secures the network path. SASE secures the access layer with Zero Trust controls.

🤖

Embedded Systems & RTOS

VxWorks, FreeRTOS, AUTOSAR, Siemens SIMATIC. Purpose-specific OS. Security breaches can cause physical harm to people.

🏭

ICS / SCADA

AIC Triad — Availability first. PLC, HMI, DCS, SCADA, data historian. Stuxnet = defining ICS attack. NIST SP 800-82.

🔒

Zero Trust Architecture

Never trust, always verify. Adaptive identity, threat scope reduction, policy-driven access. Control plane + data plane. NIST SP 800-207.

Key Terminology

Lesson 6 Glossary

Essential terms and definitions for the CompTIA Security+ SY0-701 exam.

Adaptive Identity
Continuous, context-based identity verification in Zero Trust — adjusted by what, where, when, and how access is requested.
AIC Triad
Reversed CIA Triad used in ICS/SCADA: Availability → Integrity → Confidentiality. Safety is the overriding priority.
Auto-Scaling
Automated adjustment of cloud computing resources based on real-time demand. Must inherit security configurations automatically.
CapEx / OpEx
Capital Expenditure (up-front hardware costs) vs. Operational Expenditure (pay-as-you-go cloud costs). Cloud converts CapEx to OpEx.
Container
A portable, self-contained application package with code, libraries, and dependencies. Runs consistently on any platform. Managed by Docker.
Deperimeterization
Security approach that shifts focus from defending network boundaries to protecting individual resources and data regardless of location.
Device Posture
The security status of a device — including security configurations, software versions, and patch levels. Used in Zero Trust policy decisions.
DCS
Distributed Control System — ICS that manages process automation within a single industrial site. Governed by a control server.
FaaS
Function as a Service — serverless computing where individual functions execute in response to triggers. Provider manages all infrastructure.
GRS
Geo-Redundant Storage — highest cloud replication tier. Data replicated to a geographically distant secondary region for regional disaster protection.
HCL
HashiCorp Configuration Language — used in Terraform and Consul for IaC. Supports variables and has concise, readable syntax.
HMI
Human-Machine Interface — how operators interact with PLCs in ICS environments. Can be a physical panel or software. Common ICS attack target.
IaC
Infrastructure as Code — managing computing infrastructure through machine-readable definition files (YAML, JSON, HCL). Version-controlled.
ICS
Industrial Control System — provides workflow and process automation for critical infrastructure including power, water, and national security.
Implicit Trust Zone
The encrypted, transient data pathway established between the Policy Enforcement Point and a resource for an approved Zero Trust session.
IoT
Internet of Things — the network of physical devices embedded with sensors, software, and connectivity enabling data collection and exchange.
ISA
Interconnection Security Agreement — defines security responsibilities, encryption, access controls, audit rights, and compliance between an org and its CSP.
Microsegmentation
Granting trust per individual transaction — not per session. The granular Zero Trust approach contrasting with perimeter-based persistent trust models.
Mirai Botnet
Malware that infected millions of IoT devices via default credentials and weaponized them to launch massive DDoS attacks against major websites.
Multi-Tenancy
Multiple organizations sharing the same physical cloud infrastructure with logical separation of their data and applications.
NFV
Network Functions Virtualization — running network functions (firewalls, routers) as software on VMs and containers instead of dedicated hardware.
Northbound / Southbound API
SDN interfaces. Northbound: SDN applications to controller. Southbound: controller to network appliances. North = up. South = down.
PLC
Programmable Logic Controller — embedded device that directly controls actuators and reads sensors in an ICS environment.
Policy Enforcement Point
The Zero Trust component that mediates every access request. The only process permitted to interface with the Policy Administrator.
PUE
Power Usage Effectiveness — measures data center energy efficiency. Lower PUE = more energy used for computing vs. supporting infrastructure.
RTOS
Real-Time Operating System — purpose-specific OS for applications requiring real-time processing and guaranteed response times.
SASE
Secure Access Service Edge — combines WAN technologies with cloud-based security services under a Zero Trust model for any user from any location.
SCADA
Supervisory Control and Data Acquisition — manages large-scale, multi-site ICS environments via WAN communications including cellular and satellite.
SD-WAN
Software-Defined Wide Area Network — connects distributed locations with centralized security policy, encryption, and intelligent routing. Secures the network path.
SLA
Service Level Agreement — defines performance, availability, and support commitments between a CSP and an organization. Includes penalties for non-compliance.
Stuxnet
Nation-state malware that attacked SCADA systems controlling centrifuges in Iran's nuclear program — caused real physical equipment damage. NIST SP 800-82.
Type 1 Hypervisor
Bare-metal hypervisor running directly on physical hardware — no host OS. Enterprise environments. Examples: VMware ESXi, Microsoft Hyper-V.
Type 2 Hypervisor
Hosted hypervisor running on top of a host OS. Used for dev and testing. Examples: VMware Workstation, Oracle VirtualBox.
Vendor Lock-In
Over-dependence on one cloud provider making switching extremely difficult. Mitigated through multi-cloud strategies and data portability planning.
Zero Trust Architecture
Security model assuming nothing is trusted by default — regardless of location. Every access request must be authenticated and continuously verified. Core principle: Never trust, always verify. NIST SP 800-207.
