Lesson Overview

From designing security to evaluating whether it actually works

In the previous lesson, you learned how secure networks and cloud environments are designed — the architecture, segmentation, and controls that go into building a defensible network.

This lesson makes a pivot from design to evaluation. Designing a network to be secure and knowing that it is secure are two very different things. Evaluating network security capabilities means being able to assess whether the protections you put in place are present, configured correctly, and actually doing their job.

💡 Key Idea
A control that exists on paper is not the same as a control that works. Evaluation is the discipline of asking three questions about every protection: Is it present? Is it configured correctly? Is it being monitored? A security professional who can answer those questions can find gaps, recommend fixes, and defend the network with confidence rather than assumption.

Security Baselines

Establishing a known-good standard to measure everything else against

💡 Key Idea
You cannot evaluate whether something is secure without a standard to compare it to. A security baseline is that standard — a documented, known-good set of configurations. Once you have a baseline, "secure" stops being a matter of opinion and becomes something you can measure.
📖 Definition
A secure baseline is a collection of standard configurations and settings that covers every major area of an IT environment, applied consistently and centrally across the organization.

What a Baseline Covers

A baseline is not about a single device. It defines the secure configuration for the whole environment:

  • Operating systems, network devices, and applications
  • Cloud instances and the patching and update process
  • Access controls, password policies, and encryption
  • Logging, monitoring, and endpoint protection

Why Baselines Matter

A good baseline delivers three things at once:

🛡️

Security

Enforces consistent, hardened configurations so systems are not left in weak default states.

🧭

Manageability

Gives administrators one clear standard to build and maintain systems against.

⚡

Efficiency

Removes guesswork from configuration decisions, so teams move faster with fewer mistakes.

Where Baselines Come From

Organizations do not build baselines from scratch. Trusted industry bodies and vendors have already done that work.

CIS Benchmarks

Published by the Center for Internet Security, CIS Benchmarks are among the most widely used baseline resources in the industry. They cover networks, operating systems, and applications, are updated continuously as new threats emerge, and align with major compliance frameworks.

PCI DSS NIST 800-53 SOX ISO 27000

They also include product-specific guidance for Windows desktop and server, macOS and Linux, Cisco network devices, web browsers and web servers, database and email servers, and VMware ESXi. If you need to securely configure one of those systems, CIS has a documented benchmark for it.

Two levels of CIS Benchmarks: Level 1 applies basic security that does not impact usability. Level 2 applies maximum security and may impact functionality. A payment-processing server handling credit cards would use Level 2.

STIGs (Security Technical Implementation Guides)

STIGs are a specific type of secure baseline developed by the Defense Information Systems Agency (DISA) for U.S. Department of Defense IT infrastructure. They define standardized configurations much like CIS Benchmarks, but they are written for federal and DoD environments, where they are mandatory.

Vendor Guidance

Device manufacturers and software vendors also publish secure configuration guides for their own products, which complement CIS and STIG sources.

Example
A hospital and a retail company face different threats, so they need different baselines. The hospital selects a baseline tuned to HIPAA obligations and patient-data protection, while a defense contractor is required to apply DISA STIGs. The baseline is chosen to match the organization's specific risk profile and regulatory needs — not picked off the shelf.
⚠ Watch For: Configuration Drift
Without a baseline to measure against, systems "drift" over time. Someone disables a firewall rule for testing and forgets to re-enable it. Someone grants temporary admin access that is never revoked. A baseline is what lets you detect that drift and correct it.
⚙️

Configuration Enforcement & Automation

Deploying the baseline consistently at scale, and measuring compliance

💡 Key Idea
A baseline only protects you if every system actually follows it. Configuring hundreds or thousands of machines by hand is slow and error-prone, and every missed setting is a gap. Automation enforces the baseline everywhere, every time — and catches systems that drift away from it.

Configuration Management Tools

These tools push baseline configurations automatically across many systems at once, and detect and correct deviations from the baseline as they happen. Three are widely used:

🟧

Puppet

Uses a declarative language — you define the desired state, and Puppet ensures it is maintained across large, diverse environments.

🍳

Chef

Uses a Ruby-based scripting approach, treating infrastructure as code so configuration is version-controlled and auditable.

🅰️

Ansible

Agentless — no software is installed on managed systems. Uses simple YAML playbooks, which is why it is popular for low-overhead automation.

In Windows environments, Active Directory Group Policy plays the same role, pushing standardized settings out to domain-joined machines.

Measuring Compliance with SCAP

Deploying a baseline is only half the job. You also need to verify that systems still comply with it. That is the role of the Security Content Automation Protocol.

📖 Definition
SCAP is a framework of standards that lets tools automatically assess and verify whether systems comply with a defined baseline.

SCAP-Compliant Tools

🐧

OpenSCAP

An open-source SCAP tool that assesses system configurations and generates compliance reports against a defined baseline. Widely used in Linux environments.

📊

CIS-CAT Pro

Developed by the Center for Internet Security to assess systems specifically against CIS Benchmarks, producing detailed reports of exactly where a system meets or deviates from the benchmark.

Example
A team uses Ansible playbooks to deploy a CIS baseline across 500 servers and workstations in a single run. They then schedule CIS-CAT Pro scans that report which machines have drifted out of compliance — so the gaps are found and fixed automatically rather than discovered during an incident.
Ansible Puppet Chef AD Group Policy OpenSCAP CIS-CAT Pro SCAP

Wireless Security Evaluation

Auditing the airspace: site surveys, heat maps, and rogue AP detection

💡 Key Idea
Wired networks stop at the wall. Wireless networks extend into the air, where signals travel beyond the building and anyone in range can listen or try to connect. Evaluating wireless security means continuously auditing that invisible attack surface, because it changes every time furniture moves, a new device appears, or an attacker sets up shop nearby.

The Three Core Evaluation Techniques

1

Site Survey

Walk the facility measuring signal coverage, interference, and dead zones. The survey informs where access points should be placed and at what power level.

2

Signal / Heat Mapping

Turn the survey data into a visual heat map of coverage strength. This reveals gaps in coverage and, just as important, signal "bleeding" outside the building where an attacker could pick it up.

3

Rogue AP Scanning

Scan for unauthorized access points. A rogue AP — including an evil twin that imitates a legitimate network name — creates a backdoor onto the network or lures users into connecting to an attacker.

4

Validate & Remediate

Confirm that every detected access point is authorized and correctly configured, and remove or reposition anything that is not. Then repeat the cycle on a regular schedule.

Example
A monthly site survey produces a heat map showing the office Wi-Fi signal reaching well into the parking lot — a clear exposure. The same scan flags an unknown access point broadcasting the company's network name from a nearby desk: a rogue AP planted to harvest credentials. Both findings come from routine evaluation, not from waiting for an incident.
Site Survey Heat Map Rogue AP Evil Twin

Network Authentication

Controlling who is allowed to connect in the first place

💡 Key Idea
Before a device is allowed to send traffic on the network, the user behind it should have to prove who they are. Enterprise authentication does more than say yes or no — it enriches your logs, strengthens data protection, and gives you granular control over exactly who can reach what.

RADIUS and MFA

RADIUS provides centralized authentication, authorization, and accounting — often called AAA. Instead of each device or service checking credentials on its own, RADIUS gives the organization one central place to authenticate users across the whole network, which also means one consistent log of who connected and when.

Pairing RADIUS with multi-factor authentication (MFA) raises the bar further: a stolen password alone is no longer enough to get onto the network.

The 802.1X Model

📖 Definition
802.1X is the IEEE standard for port-based network access control. A device connected to a switch port or wireless network cannot pass any traffic until it has been authenticated.

802.1X works through three roles:

Role Who It Is Function
Supplicant The connecting device (laptop, phone) Requests access and supplies credentials to get onto the network.
Authenticator The switch or wireless access point Acts as the gatekeeper for the port, holding traffic until authentication succeeds.
Authentication Server The RADIUS server Verifies the credentials and tells the authenticator whether to open the port.
Example
An employee plugs a laptop into a conference-room network jack. The switch (authenticator) blocks all traffic on that port and forwards the login to the RADIUS server. Only after the employee passes RADIUS with MFA does the switch open the port and allow the laptop onto the network.
RADIUS MFA 802.1X AAA
🚦

Network Access Control (NAC)

Checking each device's health before it is let in

💡 Key Idea
Authentication proves who you are. NAC asks a different question: is your device safe to admit? NAC is the gate-keeper that inspects each device's posture — its security health — and decides whether to grant access, block it, or send it somewhere to be fixed first.

What NAC Checks (Device Posture)

Before granting access, NAC evaluates whether the connecting device meets the organization's requirements:

  • Patch level — is the operating system up to date?
  • Antivirus — is endpoint protection installed and current?
  • Host firewall — is it enabled?
  • Configuration compliance — does the device match the security baseline?

How NAC Responds

1

Device Connects

A device requests access to the network.

2

Posture Check

NAC inspects patch level, antivirus status, and configuration against policy.

3

Decision

Compliant devices are granted access. Non-compliant devices are denied or placed in a quarantine network.

4

Remediate & Re-check

A quarantined device updates its patches or antivirus, then is re-evaluated before being admitted.

Example
A contractor connects a personal laptop with antivirus definitions three months out of date. NAC detects the outdated posture and drops the laptop into a quarantine VLAN with access only to the update server. Once the antivirus is current, NAC re-checks the device and grants it normal access.
⚠ Why It Matters
Without NAC, a single non-compliant or already-infected device can connect freely and become the foothold an attacker uses for lateral movement across the network. NAC enforces the security baseline at the moment of connection — the "gate-keeper" for the whole network.
Posture Check Patch Level Antivirus Status Quarantine VLAN
🔥

Firewalls, IDS/IPS & Web Filtering

The active monitoring and control layer of network security

💡 Key Idea
These three tool categories actively monitor and control traffic as it moves through the network. The crucial point is that no single one is sufficient on its own — each covers different threats. Used together, they form overlapping layers, so that if one layer is bypassed, another is still in place to catch what was missed.

The Three Tool Categories

🔥

Firewalls

Control what traffic is allowed to enter and leave the network, based on defined rules covering source, destination, port, and protocol.

🛰️

IDS / IPS

Monitor network and host activity for threats. An IDS detects and alerts; an IPS detects and actively blocks the malicious traffic.

🌐

Web Filtering

Blocks access to malicious and inappropriate content at the application layer, stopping users from reaching dangerous sites.

IDS vs IPS

Aspect IDS (Detection) IPS (Prevention)
Action on a threat Detects and raises an alert Detects and actively blocks it
Placement Out of band, watching a copy of traffic In line, traffic passes through it
Risk Threat may proceed before response Threat stopped automatically
Defense in Depth
Layering these controls is the principle of defense in depth. The firewall sets the rules at the border, the IPS inspects what gets through for known attack patterns, and the web filter blocks dangerous destinations. An attack would have to defeat all three to succeed.
Example
A user clicks a link in a phishing email. The web filter blocks the known-malicious site outright. On a different, not-yet-categorized site, the IPS recognizes the exploit pattern in the payload and drops the connection. Even if malware did install, the firewall's outbound rules block its attempt to call home to a command-and-control server. Three layers, three chances to catch it.
Firewall IDS IPS Web Filter Defense in Depth

Bringing It Together

Six operational guidelines working as one layered posture

💡 Key Idea
Everything in this lesson connects. These are not six separate security programs — they are one integrated, layered security posture. Evaluating network security means confirming that all six of these guidelines are present and working together.

The Six Operational Guidelines

# Guideline What It Means in Practice
1 Select a baseline based on risk Choose CIS Benchmarks, STIGs, or vendor guidance that fit the organization's environment, regulations, and threat landscape — not a one-size-fits-all standard.
2 Automate baseline rollout Deploy and enforce the baseline with Active Directory Group Policy or Ansible, Puppet, and Chef, so every system is configured correctly every time.
3 Continuously audit wireless Run regular site surveys and validate access points, scanning for rogue APs in the airspace.
4 Use enterprise authentication Authenticate users with RADIUS and MFA to enrich logging, protect data, and gain granular control.
5 Enforce access with NAC Use NAC gate-keepers to check device posture before any device is admitted.
6 Monitor with firewalls, IDS/IPS, web filtering Control and inspect traffic continuously, blocking malicious or inappropriate content as overlapping layers.

One Scenario, All Six Guidelines

Here is how the guidelines come together for a single financial-services company — one integrated posture rather than six isolated efforts:

1

Baseline tailored to regulation

The company selects a CIS Benchmark baseline tuned to its regulatory requirements.

2

Automated at scale

It uses Ansible to deploy that baseline automatically across 500 servers and workstations.

3

Wireless audited monthly

The wireless team runs monthly site surveys and scans for rogue access points.

4

Authenticated access

All users authenticate to the network using RADIUS with MFA.

5

Posture enforced

NAC checks every connecting device for patch compliance and antivirus status before granting access.

6

Continuous monitoring

Firewalls, IPS, and web filters monitor all traffic continuously, catching and blocking threats in real time.

📝 Final Lesson Takeaway
Evaluating network security capabilities is a continuous process, not a one-time audit. It needs a baseline chosen for real risk, automation that enforces it everywhere, wireless auditing of the airspace, authentication that controls who connects, NAC that checks device health, and firewalls, IDS/IPS, and web filtering that watch all traffic.

A network that is deployed but not evaluated is only assumed to be secure. Real security is measurable, visible, and continuously improving.
