Topic 3A

Cryptographic Algorithms

Before we can protect systems, we need to understand the building blocks of cryptography. This topic covers the three fundamental tools — symmetric encryption, asymmetric encryption, and hashing — and explains when and why to use each one.

Slide 4 3A

Core Cryptographic Concepts

Cryptography is the science of securing information by transforming it into an unreadable form. Before working with any cryptographic system, you need to know the foundational vocabulary — these terms appear constantly in the exam and in real security work.

PLAINTEXT

The original, readable message or data before any encryption is applied. This is what you want to protect.

CIPHERTEXT

The scrambled, unreadable output after encryption. Without the correct key, ciphertext is meaningless.

CIPHER / ALGORITHM

The specific mathematical method used to transform plaintext into ciphertext and back. AES and RSA are examples of ciphers.

KEY

The secret value used with the cipher to encrypt or decrypt. The same algorithm with different keys produces completely different outputs.

CRYPTANALYSIS

The art and science of breaking cryptographic systems — finding weaknesses or cracking ciphertext without the key.

ALICE, BOB & MALLORY

Traditional characters used in crypto scenarios. Alice sends, Bob receives, and Mallory is the malicious attacker intercepting traffic.

📌 Key Points
  • Encryption uses substitution (replace characters) and transposition (rearrange characters)
  • The cipher and the key are separate — the algorithm is usually public knowledge; the key must be secret
  • The three cryptographic tool types: Hashing, Symmetric encryption, Asymmetric encryption
Slide 5 3A

Symmetric Encryption

Symmetric encryption uses a single shared key for both encrypting and decrypting data. Think of it like a padlock where both people need a copy of the same key — fast and efficient, but the key must be shared securely beforehand.

📌 Key Points
  • One key both encrypts and decrypts — both parties must possess the same secret key
  • Fast — designed for bulk encryption of large amounts of data (files, drives, streams)
  • Core weakness: the key distribution problem — how do you share the key without it being intercepted?
  • Provides confidentiality only — since both sides have the same key, you cannot prove who encrypted the data
  • Primary modern algorithm: AES (Advanced Encryption Standard) — the gold standard, used everywhere
Think About It: BitLocker uses AES symmetric encryption to lock your entire hard drive. One key — stored in the TPM chip — encrypts and decrypts everything. If someone steals the drive, they cannot read it. But if they steal your key, they can read everything.
Symmetric encryption provides confidentiality only. It cannot prove who sent a message because both parties share the same key. This is a frequent exam distinction — know why it cannot provide non-repudiation.
Slide 6 3A

Key Length

The strength of an encryption key depends on its length. Longer keys exponentially increase the number of possible values an attacker must try — making brute force attacks impractical.

KEYSPACE

The total range of all possible key values. A 128-bit key has 2¹²⁸ possible values — a number larger than atoms in the observable universe.

BRUTE FORCE

Trying every possible key until one works. Longer keys make this computationally infeasible — AES-256 cannot be brute forced with current technology.

AES-128 vs AES-256

AES-256 is not twice as strong as AES-128 — it is 2¹²⁸ times stronger. Exponential difference, not linear.

TRADE-OFF

Longer keys require more CPU, memory, and power. Engineers balance security requirements against performance constraints.

Know that AES-256 is exponentially stronger than AES-128 — not just twice as strong. Also know that legacy ciphers DES (56-bit) and RC4 are cryptographically weak and should never be used in modern systems.
Slide 7 3A

Asymmetric Encryption

Asymmetric encryption uses a mathematically linked key pair — one public, one private. What one key encrypts, only the other can decrypt. This solves the key distribution problem because the public key can be shared freely.

PropertyPublic KeyPrivate Key
VisibilityShared openly with anyoneNever shared — kept secret
Used to encryptEncrypt messages to the key ownerCreate digital signatures
Used to decryptVerify digital signaturesDecrypt messages encrypted with public key
DerivabilityCannot derive the private key from the public key — mathematically infeasible
📌 Key Points
  • Used for small amounts of data only — mainly authentication and key exchange (not bulk data)
  • RSA: most common asymmetric cipher — requires 2,048-bit keys or better
  • ECC (Elliptic Curve Cryptography): achieves equivalent security with smaller keys (256-bit) — preferred for mobile/IoT
  • Solves the key distribution problem — your public key can be posted anywhere
SSH Example: When you SSH into a Linux server, you place your public key on the server. When connecting, the server uses your public key to verify your identity. Your private key stays on your local machine and never travels the network. This is asymmetric authentication in practice.
Slide 8 3A

Hashing

A hash function takes any input and produces a fixed-length output called a digest. Unlike encryption, hashing is a one-way process — you cannot reverse a hash to get the original input. Think of it as a fingerprint for data.

ONE-WAY

Given a hash digest, it is computationally impossible to reconstruct the original input. You can verify but not reverse.

ANTI-COLLISION

No two different inputs should produce the same digest. A "collision" is a serious vulnerability — it means two inputs look the same to the hash function.

SHA-256+

Current standard — part of the SHA-2/SHA-3 families. Produces 256-bit output. Use for passwords, file integrity, and digital signatures.

MD5 (BROKEN)

Only 128-bit output. Known collision vulnerabilities — do not use for security. Still seen for non-security checksums, but avoid for passwords.

📌 Key Points
  • Password storage: websites store the hash of your password — when you log in, they hash your input and compare, never storing plaintext
  • Integrity checking: software downloads provide a hash so you can verify the file hasn't been tampered with
  • SHA-256 (or better) for security; MD5 only for non-security checksums
Hashing ≠ Encryption. Hashing is one-way — you cannot decrypt a hash. Encryption is two-way — you can decrypt with the key. This distinction is critical on the exam.
Slide 9 3A

Digital Signatures

Digital signatures combine asymmetric encryption with hashing to solve three security problems at once: proving who sent a message, proving the message wasn't changed, and ensuring the sender cannot deny sending it.

StepSender (Alice) DoesReceiver (Bob) Does
1Hashes the document → creates a digestReceives document + signature
2Encrypts the digest with her private key → this is the digital signatureHashes the document himself → gets his own digest
3Sends document + signature to BobDecrypts Alice's signature using her public key
4—Compares the two digests — if they match, all three properties are confirmed
📌 Three Security Properties Provided
  • Integrity: if the document was altered after signing, the hashes won't match
  • Authentication: only Alice's private key could have created that signature — proves identity
  • Non-repudiation: Alice cannot deny signing — only her private key could produce that signature
Software Updates: Every Windows update is digitally signed by Microsoft using their private key. Before installation, Windows verifies the signature with Microsoft's public key. If the signature doesn't match, the update is rejected — protecting you from fake or tampered updates.
3B
Topic 3B

Public Key Infrastructure (PKI)

Asymmetric encryption is powerful, but it raises a question: how do you know a public key really belongs to who it claims to? PKI is the trust framework that answers this question — it is the backbone of HTTPS, email security, code signing, and identity on the internet.

Slide 13 3B

Certificate Authorities

If Alice sends you her public key, how do you know it's really hers? An attacker (Mallory) could intercept it and substitute her own key. PKI solves this with Certificate Authorities — trusted organizations that verify identities and vouch for public keys.

📌 Key Points
  • PKI (Public Key Infrastructure): the complete framework for managing trust around public keys
  • Certificate Authority (CA): verifies the identity of subjects and signs their certificates — like a trusted notary
  • CA maintains a repository of all issued certificates and public keys
  • Third-party CA: a commercial or public CA with established worldwide trust (DigiCert, Let's Encrypt, Sectigo)
  • Your browser includes a pre-installed trust store of root CAs — certificates from these CAs are automatically trusted
The Padlock Icon: When your browser shows a padlock on a website, it has verified that the website's certificate was signed by a CA in its trust store. Without this verification chain, HTTPS would provide encryption but no identity verification — you could be talking to an impersonator.
Slide 14 3B

Digital Certificates

A digital certificate is an official document — signed by a CA — that binds a public key to an identity. It is the digital equivalent of a government-issued ID that includes your public key.

CERTIFICATE CONTENTS

Subject identity, subject's public key, validity period (issue + expiry dates), issuing CA name, and the CA's digital signature.

X.509 (PKIX)

The international standard format for digital certificates. Almost all TLS, email, and code-signing certificates use X.509 format.

PKCS

Public Key Cryptography Standards — defines file formats for keys and certificates (e.g., .pem, .pfx, .p12 file types).

CERTIFICATE TYPES

TLS/HTTPS (web), Code-signing (software), S/MIME (email), Client authentication — each serves a different purpose.

Try this: In your browser, click the padlock icon on any HTTPS site → "Certificate is valid." Examine the fields — Subject, Issuer, Validity period, and Public key. Recognizing these fields in exam questions is a key skill.
Slide 15 3B

Root of Trust & Certificate Chain

If CAs sign certificates, who signs the CA's certificate? At the very top of the PKI hierarchy is the Root CA — a self-signed certificate that is trusted because it is pre-installed in operating systems and browsers. Everything builds down from there.

LayerWho It IsWhat It Does
Root CATop-level, self-signedPre-installed in OS/browser trust stores; signs intermediate CA certificates
Intermediate CASigned by Root CAIssues certificates to end subjects; insulates root from daily operations
Leaf CertificateIssued to a website/user/deviceThe certificate your browser or app actually sees and validates
Self-signed CertSigned by itselfNo chain of trust — useful for internal/dev use only; browsers show a warning
📌 Key Points
  • Root CAs are kept offline in maximum security — accessed only for rare signing ceremonies
  • Intermediate CAs handle day-to-day certificate issuance — protecting the root from exposure
  • Self-signed certificates can encrypt traffic but cannot prove identity — never use in production
  • Browser warning on self-signed certs = no chain of trust verified
Slide 16 3B

Certificate Signing Requests (CSR)

Getting a certificate issued follows a specific process. The most important rule: you always generate your own key pair, and your private key never leaves your control — not even to the CA.

📌 CSR Process — Step by Step
  • Step 1 — Generate Key Pair: You (the subject) generate a public/private key pair locally on your system
  • Step 2 — Create CSR: You create a Certificate Signing Request containing your public key and identity information (name, domain, organization)
  • Step 3 — Submit to CA: You send only the CSR to the CA — the private key never leaves your system
  • Step 4 — CA Verifies: The CA verifies your identity through their registration process
  • Step 5 — Certificate Issued: CA signs your CSR with its private key and returns your signed certificate
A critical exam point: the private key is NOT included in the CSR and is never sent to the CA. If your private key leaves your system, your certificate is compromised. Generate and store it locally only.
Slide 17 3B

Subject Name Attributes

Certificates must specify who or what they are issued to. There are two fields used for this — an older legacy approach and the modern standard. Know both, because you will see both in real systems.

COMMON NAME (CN)

Legacy field — historically used to store the domain name (FQDN). Technically deprecated for this use but still present in most certificates for compatibility.

SUBJECT ALTERNATIVE NAME (SAN)

Modern standard — structured field that holds domain names, IP addresses, or email addresses. Modern browsers require a properly configured SAN.

MULTI-DOMAIN (SAN CERT)

A single certificate can list multiple domains in the SAN field — covering example.com, shop.example.com, blog.example.com, etc.

WILDCARD

*.example.com in the SAN covers all subdomains of example.com — one certificate for unlimited subdomains at that level.

The SAN field MUST be configured with the FQDN for modern browsers to accept the certificate. CN alone is no longer sufficient. Wildcard certificates use *.domain.com — they cover one level of subdomains only.
Slide 18 3B

Certificate Revocation

Sometimes a certificate must be invalidated before it expires — for example, when a private key is stolen. There are two mechanisms for this, and knowing the difference is important for the exam.

MechanismHow It WorksLimitation
CRLCA publishes a batch list of all revoked/suspended certificates; clients download and checkCan be large; periodic updates mean it may be stale — revocation gap exists
OCSPClient queries CA's OCSP server in real-time for one specific certificate's statusRelies on CA server availability; some implementations cache responses
📌 Key Points
  • Revocation = permanent invalidation | Suspension = temporary hold (with a reason code)
  • CRL (Certificate Revocation List): batch list, periodically updated — can be stale
  • OCSP (Online Certificate Status Protocol): real-time single-certificate query — more current
  • OCSP is preferred for modern systems needing timely revocation status
Slide 19 3B

Key Management

Keys must be managed throughout their entire useful life — from the moment they are created until they are retired. Poor key management is one of the most common causes of cryptographic failures in real systems.

📌 Key Lifecycle Stages
  • Generation: key is created — quality of randomness (entropy) is critical
  • Storage: key must be stored securely — ideally in hardware (TPM or HSM)
  • Revocation: key can be invalidated if compromised before expiration
  • Expiration & Renewal: keys have a defined lifetime — rotation is a security requirement
  • Decentralized: each host/user stores their own key — flexible but hard to manage at scale
  • Centralized KMS: dedicated key management server — better governance, auditing, and control
  • KMIP: the interoperability protocol that allows different KMS systems to communicate
Cloud Example: AWS Key Management Service (KMS) is a centralized key management system. When you encrypt data in AWS, KMS generates, stores, and controls the encryption keys. You interact with keys only through APIs — you never see raw key material. This is enterprise-scale centralized key management.
Slide 20 3B

Cryptoprocessors & Secure Enclaves

Generating high-quality cryptographic keys requires true randomness. Standard software random number generators can be predictable. Hardware cryptoprocessors solve this by generating keys using hardware-grade entropy and storing them in tamper-resistant environments.

TPM

Trusted Platform Module: a cryptoprocessor chip built into the CPU or motherboard. Generates and stores keys securely. Required by BitLocker on Windows.

HSM

Hardware Security Module: a dedicated, removable, or appliance-based cryptoprocessor. Used by CAs and banks. Tamper-evident — wipes keys if physically breached.

SECURE ENCLAVE

A protected region of system memory that isolates cryptographic keys from the OS. Apple's Secure Enclave on iPhone is the most well-known example.

ENTROPY

The measure of randomness used in key generation. Low entropy = predictable keys = easier to crack. Hardware provides true entropy.

Know the difference between TPM (built into hardware, for general use) and HSM (dedicated appliance, for high-assurance environments like CAs). Both provide reduced attack surface and tamper-evident key storage.
Slide 21 3B

Key Escrow

What happens if a key holder is unavailable and the organization needs to recover encrypted data? Key escrow provides a backup copy of keys with a trusted third party — but this must be done carefully to avoid creating a new vulnerability.

📌 Key Points
  • Key escrow: backing up encryption keys with a trusted third party for recovery purposes
  • Risk: anyone with escrowed key access could impersonate the original key holder
  • M-of-N control: the key is split into N parts; M parts must cooperate to reconstruct — no single person can recover alone
  • Example: M=3, N=5 means any 3 of 5 key holders must participate in a key recovery operation
  • Key escrow is a deliberate trade-off between availability (can recover data) and security (limits who can access)
M-of-N control is a critical concept. On the exam: if M=3 and N=5, you need at least 3 of the 5 key parts to reconstruct the key. This prevents any single insider from performing unauthorized key recovery.

Topic 3C

Cryptographic Solutions

Now we apply everything to real security scenarios. This topic covers how to choose the right cryptographic solution for each situation — data at rest, in transit, or in use; database protection; transport security; password handling; and emerging technologies like blockchain.

Slide 24 3C

Encryption Supporting Confidentiality — Data States

Before applying encryption, you must ask: where is the data right now? The answer determines which encryption approach is appropriate. The Security+ exam tests all three data states.

Data StateDescriptionProtection Methods
At RestStored on a disk, database, or backup — not currently movingDisk encryption, file encryption, database encryption
In TransitMoving across a network from one system to anotherTLS, VPN, IPSec, HTTPS, SSH, S/MIME
In UseActively loaded in memory and being processed by an applicationSecure enclaves, trusted execution environments
📌 Hybrid Encryption — Best of Both Worlds
  • Problem: asymmetric encryption is too slow for large data
  • Solution: Data Encryption Key (DEK) — a symmetric key that encrypts the actual data (fast)
  • The DEK is then protected by a Key Encryption Key (KEK) — an asymmetric private key that encrypts the DEK
  • This is how HTTPS, encrypted email (S/MIME, PGP), and most real-world systems work
Slide 25 3C

Disk and File Encryption

Multiple levels of encryption can be applied to storage, each with different scope and implementation. Choosing the right level depends on what you are protecting and from whom.

FULL DISK ENCRYPTION

Encrypts the entire drive including the OS. BitLocker (Windows), FileVault (macOS). Protects against physical theft of the drive.

SELF-ENCRYPTING DRIVE (SED)

Encryption implemented in the drive's own firmware — transparent to the OS. The drive handles all encryption/decryption automatically.

VOLUME ENCRYPTION

Software-based encrypted containers that mount as drives (e.g., VeraCrypt). Protects specific data without encrypting the whole disk.

FILE/FOLDER ENCRYPTION

OS or application-level encryption of individual files. EFS on Windows. Granular control — protects files even from other users on the same system.

Know the distinction: full disk encryption protects against physical theft. File-level encryption provides granular access control within the system. Defense in depth uses both together.
Slide 26 3C

Database Encryption

Databases hold the most sensitive organizational data — customer records, payment information, health data. Database encryption can be applied at multiple granular levels, each providing different security properties.

LevelWhat It EncryptsUse Case
Database-level (TDE)Entire database pages as they move from disk to memoryProtects raw database files from theft — transparent to apps
Column-levelSpecific sensitive fields (e.g., SSN, credit card number)Compliance requirements; protects against privileged insider access
Record/Row-levelIndividual rows based on access policiesFine-grained access control for multi-tenant or highly regulated data
📌 Key Points
  • TDE (Transparent Data Encryption): page-level, transparent to the application layer
  • Column encryption: even a DBA cannot read encrypted columns without the decryption key
  • Required for compliance: PCI-DSS (payment cards), HIPAA (health data), GDPR (EU personal data)
Slide 27 3C

Transport Encryption & Key Exchange

When data moves across a network, it passes through many systems in between. Transport encryption ensures that even if traffic is intercepted, it cannot be read. TLS is the dominant protocol for this protection.

📌 Key Points
  • TLS (Transport Layer Security): the protocol powering HTTPS — encrypts all web, API, and many application communications
  • TLS uses hybrid encryption: asymmetric for key exchange during handshake, AES symmetric for data transfer
  • Digital envelope: symmetric session key is encrypted with asymmetric key for secure exchange
  • HMAC (Hash-based Message Authentication Code): provides integrity and authentication for in-transit messages
  • TLS 1.3 is the current standard — TLS 1.0 and 1.1 are deprecated and must not be used
PCI-DSS Compliance: In 2018, PCI-DSS mandated that all merchants using payment systems must disable TLS 1.0 and 1.1. These older versions had known vulnerabilities (POODLE, BEAST). Merchants who didn't upgrade faced non-compliance penalties — a real-world consequence of failing to use appropriate transport encryption.
Slide 28 3C

Perfect Forward Secrecy (PFS)

Even with strong TLS, there is a risk: if a server's long-term private key is compromised later, an attacker who recorded past traffic could decrypt all of it retroactively. Perfect Forward Secrecy eliminates this threat.

THE PROBLEM

An attacker records encrypted HTTPS traffic today. In the future, they steal the server's private key and use it to decrypt all recorded past sessions.

PFS SOLUTION

A unique ephemeral session key is generated for every connection and discarded immediately after. Past sessions cannot be decrypted even if the long-term key is later compromised.

DIFFIE-HELLMAN (ECDHE)

The key agreement algorithm that enables PFS. Both parties independently derive the same session key without ever transmitting it across the network.

TLS 1.3 MANDATE

TLS 1.3 requires PFS — RSA key exchange (which does not provide PFS) is entirely removed. Only ECDHE-based cipher suites are allowed.

TLS 1.3 mandates Perfect Forward Secrecy — this is a tested exam point. PFS is implemented via Ephemeral Diffie-Hellman (ECDHE). Key agreement (Diffie-Hellman) is different from key exchange — the session key is derived, never transmitted.
Slide 29 3C

Salting & Key Stretching

Even when passwords are hashed properly, weak human-chosen passwords create vulnerabilities. Salting and key stretching are two complementary techniques used to dramatically increase the cost of cracking password hashes.

TechniqueHow It WorksAttack It Defeats
SaltingAdds a unique random value to each password before hashing — stored alongside the hashRainbow table attacks / precomputed hash lookups
Key StretchingApplies thousands of rounds of hashing — deliberately slowBrute force attacks — each guess costs much more compute
bcryptCombines salting + stretching with a configurable cost factorBoth rainbow tables and brute force — the gold standard for passwords
PBKDF2Password-Based Key Derivation Function 2 — applies many HMAC rounds with a saltBoth attacks — widely used in FIPS-compliant environments
📌 Key Points
  • Human passwords have low entropy — they are predictable and vulnerable to dictionary attacks
  • Salt is stored in the database with the hash — it does not need to be secret, just unique per user
  • Use bcrypt, scrypt, or PBKDF2 for password storage — never plain SHA-256
Slide 30 3C

Blockchain

Blockchain is a distributed ledger technology that uses cryptographic hashing to create tamper-proof, permanent records. While most famous for cryptocurrency, it has significant security applications in any domain requiring trustworthy, unalterable records.

📌 Key Points
  • An expanding list of blocks, each containing a hash of the previous block — altering any block breaks the chain
  • Immutable: transactions cannot be deleted or reversed — permanent record of all events
  • Decentralized: no single point of control or attack — distributed across many peer nodes
  • Open ledger: transactions are public and verifiable by any participant
  • Security uses beyond crypto: supply chain verification, identity management, secure voting, notarization, tamper-evident audit logs
Walmart's Food Safety: Walmart uses blockchain to track leafy greens through the supply chain. In a 2018 E. coli outbreak, they traced contaminated food from store shelf to farm source in 2.2 seconds — versus 7 days with paper records. Blockchain's immutability and traceability made this possible.
Slide 31 3C

Obfuscation Techniques

Obfuscation techniques make data unintelligible or unrecognizable without necessarily using traditional encryption. These methods appear in both defensive security controls and in how attackers try to hide malicious activity.

STEGANOGRAPHY

Hiding a secret message inside an innocent-looking carrier file (image, audio). Unlike encryption, it hides the existence of the message, not just its content.

COVERT CHANNEL

Using steganography to secretly exfiltrate data through a legitimate-looking channel — a common attacker technique to evade DLP tools.

DATA MASKING

Replacing real sensitive data with realistic but fake data. Used in test/development environments so developers can work with realistic data without seeing real customer information.

TOKENIZATION

Replacing sensitive data with a random token. The real value is stored in a separate token vault — the token alone is useless to attackers. Reversible only through the token server.

DE-IDENTIFICATION

Removing personally identifiable information (PII) from a dataset so individuals cannot be identified. Common in healthcare research and data analytics.

Know the difference: steganography hides that data exists. Tokenization replaces real data with a useless substitute (reversible via token server). Data masking replaces real data with fake but realistic data (used in dev/test). De-identification removes PII entirely.

What You Have Learned

🔒

Symmetric Encryption

One key encrypts and decrypts. Fast, for bulk data. AES-256 is the standard. Key distribution is the core challenge.

🗝️

Asymmetric Encryption

Public/private key pair. Public key shared freely, private key never leaves you. Solves key distribution. RSA and ECC.

#️⃣

Hashing

One-way, fixed-length digest. SHA-256+ for security, never MD5. Used for passwords and integrity verification.

✍️

Digital Signatures

Sign with private key, verify with public key. Provides integrity, authentication, and non-repudiation together.

🏛️

PKI & CAs

Framework for trusting public keys. CA verifies identities and issues X.509 certificates. Chain of trust: Root → Intermediate → Leaf.

📝

CSR Process

Generate your own key pair locally. Send public key + identity in CSR. CA verifies and signs. Private key stays with you always.

🚫

Revocation

CRL for batch lists; OCSP for real-time status. Revoke when private key is compromised or certificate is no longer valid.

💾

TPM / HSM / Enclave

Hardware-based key protection. TPM built into hardware; HSM dedicated appliance. Both provide tamper-evident storage.

📊

Data States

At rest (disk encryption), in transit (TLS), in use (secure enclaves). Choose the right control for each state.

🔄

Perfect Forward Secrecy

Ephemeral session keys via ECDHE. TLS 1.3 mandates PFS. Past sessions cannot be decrypted even if the private key is later stolen.

🧂

Salting & Stretching

Salt defeats rainbow tables. bcrypt/PBKDF2 stretching defeats brute force. Never store plain hashes without both protections.

🎭

Obfuscation

Steganography hides data exists. Tokenization substitutes with token. Data masking uses fake data. De-identification removes PII.
