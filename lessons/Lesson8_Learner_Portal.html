8A

Topic 8A — Device & OS Vulnerabilities

Operating systems are one of the most critical components of any infrastructure. Vulnerabilities in an OS — introduced through design flaws, coding errors, or insecure defaults — can lead to significant problems when successfully exploited.

8AIntroduction

Vulnerability Management — Introduction

Vulnerability management encompasses identifying, evaluating, treating, and reporting security vulnerabilities across operating systems, applications, and other IT components. It is a continuous cycle — not a one-time project.

🔍
IdentifyingPhase 1
Discover vulnerabilities across networks, operating systems, applications, and cloud environments using automated scanning tools, manual reviews, and threat intelligence feeds.
📊
EvaluatingPhase 2
Analyze and validate what was found. Classify vulnerabilities by type, severity, and affected asset. Determine which findings are genuine risks versus false positives.
🔧
Treating (Remediating)Phase 3
Patch, harden, upgrade, isolate, or accept risk depending on the vulnerability and the asset. Prioritize based on impact, exploitability, and asset value. Apply compensating controls when immediate remediation is not possible.
📋
ReportingPhase 4
Document findings, actions taken, and remaining exposure. Communicate status to management, compliance teams, and stakeholders. Use CVSS scores for standardized severity communication.
💡
CONTINUOUS CYCLE

After remediation — scan again to confirm the fix worked. New vulnerabilities emerge daily — the cycle restarts continuously. The goal is to reduce the window of time between vulnerability discovery and resolution.

8AOS Vulnerabilities

Operating System Vulnerabilities

Every major operating system has a vulnerability history. The significance of an OS vulnerability depends on how widely it is deployed, how easily it can be exploited, and what an attacker can do once they exploit it.

W
Microsoft Windows — Largest Install Base = Highest Target Value
Common vulnerabilities: buffer overflows, input validation failures, privilege escalation flaws. Real-world examples: MS08-067 (Conficker worm — remote code execution via crafted network packet, 2008) and MS17-010 / EternalBlue (SMB protocol vulnerabilities exploited by WannaCry ransomware, May 2017 — installed programs, modified data, or created admin accounts across networks worldwide). Know: unpatched systems = open door.
M
Apple macOS — UNIX-Based Architecture, Growing User Base
Common vulnerabilities: access control weaknesses, secure boot issues, third-party software flaws. macOS perceived as "safer" — this leads to complacency. Shellshock (2014) — a flaw in the Bash shell affecting all UNIX-based systems including macOS — allowed attackers to execute arbitrary commands with no special privileges required.
L
Linux — Dominant Server and Cloud OS
Open-source: vulnerabilities are visible to everyone, including attackers. Common issues: kernel vulnerabilities, misconfigurations, unpatched systems. Heartbleed (2014) — a flaw in the OpenSSL cryptographic library running on most Linux servers — allowed attackers to read memory contents including private encryption keys, user credentials, and session tokens.
A
Android — Most Widely Deployed Mobile OS
Open-source like Linux. Critical additional challenge: fragmentation — hundreds of device models, different manufacturers, inconsistent patch schedules. Stagefright (2015) — a vulnerability in Android's Stagefright media library present on virtually all devices — attackers could execute malicious code by sending a specially crafted MMS message. No user interaction required on some versions.
i
iOS — Closed Ecosystem, Nation-State Target
More restricted than Android — exploits are rare and extremely valuable on the black market. Watering hole attacks (2019) discovered by Google's Project Zero — nation-state attackers exploited multiple iOS vulnerabilities in a chain. Victims visited a malicious website — no further interaction required — giving attackers full device access including messages, emails, photos, location data, and credentials.
⚠️
EMBEDDED AND IOT

Every embedded system and IoT device runs a specialized OS — each introducing its own vulnerabilities. These are frequently omitted from vulnerability management programs. Devices often cannot be patched after deployment and run unsupported software versions indefinitely.

8AVulnerability Types

Vulnerability Types — Legacy, EOL, Firmware & Virtualization

LEGACY SYSTEMS

Outdated but Operational

Still in use despite shortcomings. Often incompatible with modern security tools. Replacing them is expensive and operationally risky. The vendor may still support them — but they represent outdated methods.

END-OF-LIFE (EOL)

No More Patches — Ever

Manufacturer publicly declared no further support. No updates, no security patches. New vulnerabilities discovered after EOL are permanently unaddressed. Windows 7 and Server 2008 reached EOL January 2020.

FIRMWARE VULNERABILITIES

Below the OS Layer

Meltdown & Spectre (2018) — processor-level vulnerabilities affecting virtually all computers. LoJax (2018) — UEFI firmware vulnerability enabling persistence even after full hard drive replacement or OS reinstallation. Cannot be removed by standard methods.

VM ESCAPE

Breaking Virtualization Isolation

Attacker breaks out of a VM and gains access to the host or other VMs. Cloudburst (CVE-2009-1244) — VMware ESX Server flaw allowed guest OS to execute code on the host OS. One compromised VM potentially equals all VMs on that physical host.

RESOURCE REUSE

Data Leakage Between VMs

When VMs are deleted and new ones allocated the same disk space or memory without sanitization — sensitive data from the previous VM can be recovered. Mitigation: data sanitization, encryption throughout the VM lifecycle, and resource segregation by security level.

HYPERVISOR VULNERABILITIES

Foundation-Level Attacks

Hypervisor management interfaces (used by administrators to control VMs) are potential attack vectors if authentication is weak, encryption is absent, or communication protocols contain vulnerabilities. Hypervisors require regular patching like any other software.

⚠️
EOL RISK IN SECONDARY MARKETS

Many devices remain on sale in secondhand, recertified, or reconditioned marketplaces with known severe firmware vulnerabilities and no possibility of vendor support. Organizations must vet all hardware sources — not just primary vendors.

8AZero-Day

Zero-Day Vulnerabilities

Zero-day vulnerabilities are previously unknown software or hardware flaws that attackers can exploit before developers become aware of them. The name signifies that developers have zero days to fix the problem once the vulnerability becomes known.

🚫
No Patch Exists
The vendor cannot fix what they do not know is broken. Traditional security measures are largely ineffective — antivirus relies on known signatures, firewalls rely on known attack patterns. Neither works against a zero-day.
👁️
Stealthy and Unpredictable
Systems are compromised without triggering standard detection. Damage accumulates before anyone realizes an attack is underway. The window of exposure from discovery to patch can span days, weeks, or months.
💰
Enormous Financial Value
A zero-day exploit for a mobile OS can be worth millions of dollars. Attackers will not waste a zero-day on low-value targets. Zero-days are reserved for high-value operations against government institutions, major corporations, and critical infrastructure.
🏛️
Government Stockpiling
State security and law enforcement agencies are known to stockpile zero-days to facilitate investigations. When stockpiles are compromised, adversaries gain the same capabilities — EternalBlue was originally developed and stockpiled by the NSA before being leaked and used in WannaCry.
🤝
Responsible Disclosure
Ethical researchers who discover a zero-day privately notify the vendor before public disclosure — giving time to develop a patch. Only after the patch is available (or a defined time window passes) is the vulnerability publicly disclosed. This limits potential harm and gives defenders a head start.
8AMisconfigurations

Misconfiguration Vulnerabilities

Misconfiguration is one of the most common — and most preventable — causes of security vulnerabilities. The organization creates the vulnerability itself through improper setup or management. No attacker required.

1
Default Configurations — Hardware and Network Devices
Routers, switches, and network devices ship with well-known default credentials (admin/admin, admin/password) and vulnerable management protocols (Telnet instead of SSH). An attacker scanning for default credentials finds them within minutes using automated tools.
2
Default Configurations — Software
Applications ship with debug modes enabled in production, overly permissive access controls, and sample files revealing software versions. Default administrator accounts that cannot be disabled without deliberate action are extremely common.
3
Default Configurations — Cloud Services
Cloud defaults frequently leave storage buckets and compute instances publicly accessible. An S3 bucket configured incorrectly exposes all contents to anyone on the internet. Misconfigured cloud storage buckets have been responsible for some of the largest data breaches in recent years.
4
Support and Troubleshooting — The Hidden Risk
A support technician temporarily disables a firewall rule or loosens access controls to isolate a problem — and never reverts the change. High-impact outage response bypasses change management entirely. Changes are made without documentation, testing, or approval — creating misconfigurations that surface later as breaches.
🚨
PREVENTION CHECKLIST

Change default credentials immediately. Disable unnecessary services and ports. Apply security baselines before deployment. Enforce change management discipline even during emergencies. Every undocumented change cannot be reviewed, audited, or reverted.

8ACryptographic Vulns

Cryptographic Vulnerabilities

Cryptography is the backbone of secure communication. When the cryptography itself is vulnerable — everything built on top of it is at risk. Secure communications become readable, authentication becomes bypassable.

🔢
Weak Hash FunctionsBROKEN
MD5 and SHA-1 are now considered insecure — vulnerable to collision attacks (two different inputs producing the same hash output). Never use either for password protection or digital signatures. SHA-256 or higher is required.
🔑
Weak Symmetric Encryption — DES and 3DESDEPRECATED
DES (56-bit key) — proven vulnerable to brute force attacks in the 1990s. Replaced by 3DES, which was then found vulnerable to the Sweet32 birthday attack (CVE-2016-2183, August 2016). NIST officially deprecated 3DES in 2017, recommended discontinuation by 2023. AES is the current standard.
📡
Protocol Vulnerabilities — Heartbleed, KRACKREAL ATTACKS
Heartbleed (2014) — flaw in OpenSSL implementation allowing attackers to read server memory including private keys and credentials. KRACK — Key Reinstallation Attacks in WPA2 protocol — allowed interception and decryption of Wi-Fi traffic.
🛡️
Cipher Suite Attacks — BEAST and POODLESSL/TLS
BEAST (Browser Exploit Against SSL/TLS) and POODLE (Padding Oracle On Downgraded Legacy Encryption) both exploited implementation flaws in SSL and early TLS cipher suites. POODLE forced browsers to downgrade to SSL 3.0, then exploited its known weaknesses.
🔐
Protecting Cryptographic Keys — Kerckhoffs's PrincipleBEST PRACTICE
A cryptosystem should be secure even if everything except the key is public knowledge. Keys must be generated with strong randomness, stored in HSMs or KMS, protected with access controls, and rotated periodically. A strong algorithm with a weak or exposed key is completely useless.
8AMobile Vulnerabilities

Sideloading, Rooting & Jailbreaking

ROOTING

Android

Gaining root (administrative) access on an Android device. Bypasses OS security boundaries. Malicious apps gain access to system files and corporate data. MDM platform controls may be bypassed. Device may no longer receive official security patches.

JAILBREAKING

iOS

Removing Apple's iOS restrictions. Violates Apple's terms of service — the device loses manufacturer support permanently. No security patches, no bug fixes. Apple's sandboxing and code signing mechanisms — which prevent unauthorized code execution — are removed entirely.

SIDELOADING

Both Platforms

Installing apps outside official stores (Google Play, Apple App Store). Android: APK files from third-party sources including F-Droid. iOS: typically requires jailbreaking first. Bypasses the security review process — malicious apps disguised as legitimate software can be installed with no vetting.

EXCESSIVE PERMISSIONS

App-Level Risk

Apps requesting more permissions than their purpose requires. A flashlight app requesting contacts, location, and microphone has no legitimate reason. Users routinely approve all permissions without reading them — each granted permission is a potential exploitation avenue.

🔑 Organizational Mitigations
  • MDM platforms — detect and restrict rooting, jailbreaking, and sideloading on managed devices
  • Strict organizational policy — especially in healthcare, finance, and government — with clear consequences for violations
  • Employee education — employees must understand what these practices do and why they are prohibited
  • Organizations in regulated industries should implement policies prohibiting all three practices due to data breach and compliance violation risk

8B

Topic 8B — Application & Cloud Vulnerabilities

Web applications and cloud environments are accessible over the internet — reachable by any attacker from anywhere. A single insecure line of code or a single misconfigured permission can expose an entire database or an entire organization.

8BEvaluation Scope

Evaluation Scope & Seven Scope Practices

Evaluation scope refers to the product, system, or service being analyzed for potential security vulnerabilities. Before any assessment begins — the scope must be precisely defined. Everything within the boundary gets rigorous testing. Everything outside is explicitly excluded.

🔬
Security Testing
Conducting vulnerability assessments and penetration testing to identify potential weaknesses, vulnerabilities, and misconfigurations. The most direct form of evaluation — actively probing the target for exploitable conditions.
📄
Documentation Review
Reviewing design specifications, architecture diagrams, security policies, and procedures. Verifies the system was implemented according to secure design principles. Gaps between documentation and reality are findings in themselves.
💻
Source Code Analysis
Analyzing application source code to uncover input validation failures, insecure coding practices, hardcoded credentials, and coding standard violations. Static analysis (without executing the code) or manual review by a security analyst.
⚙️
Configuration Assessment
Evaluating configuration settings against security best practices — access controls, encryption settings, authentication mechanisms, and other security-related configurations. Identifies deviations from security baselines.
🔐
Cryptographic Analysis
Assessing cryptographic mechanisms — encryption algorithms, key management practices, and secure key storage. Verifies proper implementation according to industry standards. A strong algorithm implemented incorrectly is still a vulnerability.
✅
Compliance Verification
Verifying adherence to GDPR, HIPAA, PCI-DSS, ISO 27001, SOC 2, and other applicable standards. Compliance verification does not guarantee security — but establishes a documented baseline with regulatory deadlines attached.
🏛️
Security Architecture Review
Evaluating security design to identify: insufficient segregation of duties, lack of audit trails, inadequate access controls, and gaps in security controls. Architectural weaknesses are often the most difficult to remediate — they require redesign, not just configuration changes.
8BWeb App Attacks

Web Application Attacks — XSS & SQL Injection

Web applications are accessible over the internet — reachable by any attacker from anywhere. HTTP is stateless, requiring session management through cookies and session IDs. Improper session management creates exploitable conditions across many attack types.

XSS
Cross-Site Scripting — Three Types
Reflected (Non-Persistent): Malicious input in a crafted link — the trusted site returns a page containing the injected malicious code, which executes in the victim's browser with the trust level of that site.

Stored (Persistent): Malicious code inserted into a back-end database or CMS. Every user who views the content has the malicious script execute automatically. A single payload can affect thousands of users.

DOM-Based: Targets client-side scripts that manipulate the Document Object Model. The server never sees the malicious payload — it executes entirely in the browser. Particularly difficult to detect server-side.
SQLi
SQL Injection — Adding Malicious Code to Database Queries
Web applications use SQL (SELECT, INSERT, DELETE, UPDATE) to interact with databases. In a SQLi attack, the attacker adds malicious SQL code to input the application accepts — the database executes the attacker's code, not just the intended query.

Example: Normal query: SELECT * FROM users WHERE username = 'Bob'
SQLi input: ' or 1=1# → Returns ALL users because 1=1 is always true and # comments out the rest.

Impact: Extract entire databases, insert or delete records, or execute arbitrary code with database-level privileges.
CSRF
Cross-Site Request Forgery — Exploiting Session Trust
CSRF exploits the web's inherent trust in requests that appear to come from authenticated users. A user logged into their banking portal visits a malicious site — which silently submits a funds transfer request using the user's active session. The bank processes it because the request carries valid session credentials. The user had no intention of making that request.
🚨
ROOT CAUSE

XSS exploits insufficient input validation and output encoding. SQLi exploits insufficient input sanitization before passing data to the database. Both are the result of insecure coding practices that can be prevented during development — they are not inevitable.

8BCloud-Based Attacks

Cloud-Based Application Attacks & CASB

🔀
Side-Channel AttacksCLOUD-SPECIFIC
An attacker provisions an instance on the same physical server as the victim. Shared physical resources — CPU cache, memory, storage — become an information leakage path. Data from the victim's instance is extracted through observation of shared resource behavior. Largely specific to cloud multi-tenant environments.
⛏️
CryptojackingCLOUD-SPECIFIC
Attacker gains access to a cloud account or resource and deploys cryptocurrency mining software. The mining runs continuously — consuming the victim's CPU, memory, and network resources. The victim pays the cloud bill while the attacker earns cryptocurrency. Discovery often comes through an unexpected spike in cloud costs, not a security alert.
🎣
Cloud as an Attack Platform
Attackers use cloud infrastructure to conduct attacks against others. Fraudulent phishing websites on legitimate cloud services — the cloud provider's domain and SSL certificate lend credibility. Malware hosted in cloud storage and distributed via phishing emails — recipients trust a link to a recognized cloud storage domain.
🛡️ Cloud Access Security Broker (CASB)
  • What it is — enterprise management software mediating access to cloud services by users across all device types
  • Single sign-on enforcement — extends enterprise authentication and authorization to cloud providers
  • Malware scanning — scans traffic and files before they reach cloud services or end users
  • User and resource activity monitoring — visibility into what users are doing across all cloud services, creating audit trails
  • Data exfiltration prevention — prevents access to unauthorized cloud services from managed devices
  • Three implementation modes: Forward proxy (client edge — real-time inspection), Reverse proxy (cloud edge — no device config required), API-based (synchronizes identity and access changes between enterprise and cloud)
8BSupply Chain

Software Supply Chain Vulnerabilities & SBOM

Software supply chain vulnerabilities are risks introduced at any stage of development, distribution, and maintenance. An organization's security is only as strong as the weakest link in its supply chain.

SERVICE PROVIDERS

Cloud & Dev Agencies

Inadequate security at the provider level or insecure communication between provider and supply chain. A compromised development or testing platform can inject malicious code into software before it ships.

HARDWARE SUPPLIERS

Firmware & Drivers

Hardware arriving with preinstalled firmware containing known vulnerabilities. Physical tampering during manufacturing or shipping. Drivers from unreliable providers. For IoT/embedded systems — hardware suppliers often provide the entire software stack.

SOFTWARE PROVIDERS

Libraries & Frameworks

Vulnerable or outdated third-party libraries. A compromised open-source project injecting malicious code into a legitimate update. The SolarWinds attack: trusted software update delivered malware to thousands of organizations simultaneously.

📋 Software Bill of Materials (SBOM)
  • What it is — a comprehensive inventory of every component in a software product: primary code, all dependencies, libraries, frameworks, and third-party components with names, versions, and supplier information
  • Think of it as — a nutrition label for software: lists every ingredient just as a food label lists every component
  • Enables rapid response — when Log4Shell was disclosed in December 2021, organizations with accurate SBOMs identified exposure within hours; those without spent days or weeks manually searching codebases
  • Tools: OWASP Dependency-Check (SCA tool, identifies vulnerable dependencies), OWASP Dependency-Track (continuous monitoring), SPDX standard, CycloneDX standard
OWASP Dependency-Check OWASP Dependency-Track SPDX CycloneDX National Vulnerability Database (NVD) Vendor Security Advisories

A web application's login form passes user input directly to a database query without sanitization. An attacker enters: ' or 1=1# in the username field. What is the likely result?

8C

Topic 8C — Vulnerability Identification Methods

Finding vulnerabilities before attackers do requires systematic, automated, and human-powered methods working together. Vulnerability scanning, threat feeds, OSINT, and penetration testing each reveal different aspects of the same attack surface.

8CVulnerability Scanning

Vulnerability Scanning

Vulnerability scanning is a fundamental component of every vulnerability management program — using automated tools to identify and evaluate potential security issues across the entire environment. Keeping scanner databases current is as important as running the scan itself.

🌐
External ScanningOutside-In
Conducted from outside the network — simulating an internet-based attacker. Reveals what is exposed to the public internet: open ports, externally accessible services, internet-facing vulnerabilities. Shows exactly what attackers see.
🔒
Internal ScanningInside-Out
Conducted from inside the network. Reveals what an attacker or malicious insider with internal network access could exploit. Identifies vulnerabilities not visible from outside — misconfigurations, internal services, lateral movement paths.
📊
Vulnerability Feed / Plug-ins / NVTs
Scanners depend on continuously updated vulnerability databases. Nessus calls them plug-ins. OpenVAS calls them NVTs (Network Vulnerability Tests). Without current feed updates — the scanner cannot detect recently disclosed vulnerabilities. Feed currency is as critical as scan frequency.
Tenable Nessus Greenbone OpenVAS Nmap OpenSCAP Lansweeper ManageEngine SolarWinds
⚠️
SENSITIVITY WARNING

Vulnerability scan reports are among the most sensitive documents an organization produces — they list every known weakness. Scanning tools must only run from authorized hosts. Reports must only be distributed to authorized user accounts. A stolen vulnerability scan report gives an attacker a detailed attack roadmap.

8CScan Types & Tools

Credentialed vs. Non-Credentialed Scans & Application Scanners

NON-CREDENTIALED

Unauthenticated View

Scanner probes without logging into the OS or application. Sees only what an unprivileged user on the network sees: open ports, default passwords, publicly accessible vulnerabilities. Best for: external perimeter assessment and web application scanning from an unauthenticated perspective.

CREDENTIALED

Authenticated Deep View

Scanner logs in with provided credentials. Sees internal system configurations, installed software versions, security settings, and patch levels. Most accurate and complete findings. Simulates what an insider threat or attacker with a compromised account could achieve. More intrusive.

STATIC ANALYSIS

Code Without Execution

Reviews application source code without executing it. Finds vulnerabilities in the code itself: unvalidated inputs, hardcoded credentials, insecure coding patterns. Can be integrated into development pipelines to catch issues before deployment.

DYNAMIC ANALYSIS

Testing Live Applications

Tests the application while it is running. Sends inputs and observes how the application responds. Finds broken access controls, SQL injection, XSS, and session management weaknesses. More closely simulates real attacker behavior against a live application.

PACKAGE MONITORING

Supply Chain Tracking

Continuously tracks third-party packages, libraries, and dependencies for newly disclosed vulnerabilities. SCA tools (OWASP Dependency-Check, Dependency-Track) compare software inventory against NVD and vendor advisories. Alerts when a used component becomes vulnerable.

8CThreat Feeds & OSINT

Threat Feeds, Information Sharing & OSINT

TF
Threat Feeds — Real-Time Intelligence
Continuously updated sources of information about emerging threats and newly discovered vulnerabilities. Three types of output: Behavioral Threat Research (narrative TTPs from primary research), Reputational Intelligence (lists of malicious IPs/domains and malware signatures), and Threat Data (machine-readable CTI that correlates with SIEM events). Examples: IBM X-Force Exchange, Mandiant FireEye, Recorded Future, AlienVault OTX, MISP.
IS
Information-Sharing Organizations — ISACs and CTA
Collaborative groups that exchange threat data across organizational boundaries. ISACs (Information Sharing and Analysis Centers) are sector-specific: FS-ISAC (financial), H-ISAC (healthcare), E-ISAC (electricity), A-ISAC (aviation). Cyber Threat Alliance (CTA) — nonprofit members share high-quality threat intelligence in real time with mandatory reciprocal sharing. Collective defense is stronger than individual defense.
OS
Open-Source Intelligence (OSINT)
Collecting and analyzing publicly available information to support security decisions. Sources: security blogs, forums (including underground hacking forums), social media, code repositories, dark web. Adds urgency to vulnerability management — a vulnerability being actively discussed in hacking forums moves to the top of the priority list before it appears in formal channels.

OSINT Tools: Shodan (internet-connected devices), Maltego (relationship visualization), Recon-ng (web reconnaissance framework), theHarvester (emails, subdomains, employee names from public sources).
Shodan Maltego Recon-ng theHarvester OSINT Framework IBM X-Force Exchange AlienVault OTX MISP
8CDeep & Dark Web

Deep Web, Dark Net & Dark Web

🌐
Deep Web
Any part of the World Wide Web not indexed by standard search engines. Includes pages requiring registration or login (online banking, email), pages blocking search engine indexing, and unlinked pages. Most of the deep web is ordinary private content — not inherently illegal or malicious.
🕸️
Dark Net
A network established as an overlay to internet infrastructure using specialized software that anonymizes usage. TOR (The Onion Router) — routes traffic through multiple volunteer relay nodes, each knowing only the previous and next hop. Each relay decrypts one layer of encryption. The origin and destination are concealed from any third party. Also: Freenet and I2P.
🔍
Dark Web as Threat Intelligence Source
Investigating dark web sites and message boards reveals: stolen data being traded, hacking tools and ransomware-as-a-service platforms, attack planning discussions, and zero-day vulnerabilities being auctioned. Law enforcement agencies use the same tools to infiltrate these environments. As infiltration improves — adversaries create new networks and identity verification methods to detect law enforcement.
✅
Legitimate Dark Web Uses
Privacy and anonymity for whistleblowers, journalists, activists, and individuals under repressive governments. Access to censored information in countries with strict internet censorship. Cybersecurity research — monitoring threat actor activity and emerging tools. Note: Participating in illegal activities on the dark web is strictly prohibited. All dark web research must follow legal and ethical guidelines with proper authorization.
8CPen Testing

Penetration Testing, Bug Bounties & Auditing

UNKNOWN ENVIRONMENT

Previously: Black Box

The tester has no privileged information about the network or its security systems. Extensive reconnaissance is required before any exploitation can begin. Simulates the behavior of an external attacker who starts with no inside knowledge. Most realistic.

KNOWN ENVIRONMENT

Previously: White Box

The tester has complete access to all information — architecture diagrams, source code, credentials, system documentation. No reconnaissance required. Simulates the behavior of a privileged insider threat. Most thorough — nothing is hidden from the tester.

PARTIALLY KNOWN

Previously: Gray Box

The tester has some information but not complete knowledge. Partial reconnaissance required. Simulates an attacker who has gained some internal knowledge through phishing, credential theft, or a compromised insider account. Balances realism and efficiency.

BUG BOUNTIES

Global Community Testing

Organizations offer financial rewards to independent security researchers who find and report vulnerabilities. Open to a global community — no defined time frame, ongoing and continuous. Discovers complex or obscure vulnerabilities that structured pen tests may miss. Many organizations use both pen testing and bug bounties simultaneously.

COMPLIANCE AUDIT

Regulatory Assessment

Assesses adherence to GDPR, HIPAA, PCI-DSS. Many regulations require regular penetration testing — PCI DSS mandates annual pen testing for organizations handling cardholder data. Pen test findings serve as evidence of due diligence during compliance reviews.

Topic 8D — Vulnerability Analysis & Remediation

Finding vulnerabilities is only the beginning. Analysis determines what to fix first — and why. Remediation eliminates or reduces the risk. Validation confirms the fix actually worked. Together, these stages complete the vulnerability management cycle.

8DCVE, NVD, SCAP & CVSS

Common Vulnerability Scoring — CVE, NVD, SCAP & CVSS

# CVE Identifier Format
CVE-YYYY-####

# Example: EternalBlue / WannaCry SMB vulnerability
CVE-2017-0144 → MS17-010, SMB, Remote Code Execution

# Example: Heartbleed OpenSSL vulnerability
CVE-2014-0160 → OpenSSL, Memory Read, Data Exposure

# YYYY = year discovered | #### = sequential order number (min 4 digits)
CVE
Common Vulnerabilities and Exposures — cve.mitre.org
A standardized dictionary of vulnerabilities in published operating systems and applications. Maintained by MITRE. Each CVE entry includes: identifier (CVE-YYYY-####), brief description, reference URLs, and creation date. Provides a universal reference point — any tool, team, or organization discussing the same vulnerability uses the same identifier.
NVD
National Vulnerability Database — nvd.nist.gov
Maintained by NIST. Consumes CVE entries and supplements them with: detailed vulnerability descriptions, CVSS severity ratings, affected software versions, and mitigation measures. The NVD is the authoritative repository — CVE provides identification, NVD adds analysis and remediation guidance.
SCAP
Security Content Automation Protocol — scap.nist.gov
A NIST framework for automating vulnerability management. Two primary functions: distributing vulnerability feed updates to scanners, and comparing actual system configurations against target secure baselines. Provides common identifiers so different products refer to the same vulnerability consistently — enabling automated correlation across tools.
CRITICAL9.0 – 10.0
Remote exploitation, no user interaction — immediate action required
HIGH7.0 – 8.9
Significant impact — prioritize for rapid remediation
MEDIUM4.0 – 6.9
Meaningful risk — schedule remediation within defined timeline
LOW0.1 – 3.9
Limited impact — remediate as resources allow
NONE0.0
Informational — monitor and document
ℹ️
CVSS IS A BASELINE — NOT A FINAL DECISION

CVSS provides a standardized technical severity score. It does not capture real-world exploitation activity, asset value, or organizational context. A Critical score on a test system may rank lower than a High score on a critical production database being actively exploited. Prioritization requires combining CVSS with threat intelligence and asset classification.

8DFalse Positives & Log Review

False Positives, False Negatives & Log Review

FALSE POSITIVE

Incorrect Finding

A scanner incorrectly identifies a vulnerability that does not actually exist. Example: a port flagged as a security risk based on malware association — but that port is not actually open on the system. Excessive false positives cause alert fatigue — analysts stop trusting scan results entirely.

FALSE NEGATIVE

Missed Vulnerability

A real vulnerability exists but the scanner does not detect it. Automated scripts cannot replicate human creativity. Complex vulnerabilities — chained flaws, logic errors, design weaknesses — are frequently missed. Creates a false sense of security more dangerous than knowing a gap exists.

REDUCING FALSE POSITIVES

Scan Accuracy

Active and intrusive scanning modes are more accurate — they actually test and confirm vulnerability conditions rather than pattern-matching. Credentialed scanning produces more accurate results with fewer false positives. Every flagged finding should be validated before remediation resources are committed.

REDUCING FALSE NEGATIVES

Comprehensive Coverage

Run repeat scans periodically. Use scanners from multiple vendors simultaneously — different tools use different detection approaches. Supplement automated scanning with human penetration testing. Keep scanner feeds absolutely current — the most recent CVEs require the most current definitions.

📋 Log Review — Validating Scan Findings
  • System and network logs provide independent evidence to validate or refute scan findings
  • Example: A scanner flags a running process as potentially causing OS crashes. A review of Windows event logs reveals multiple entries showing the process has failed repeatedly over the past weeks — along with subsequent failures of related processes. The log evidence confirms the finding is real and actively causing harm.
  • Log review confirms true positives, identifies false positives, reveals active exploitation impact, and supports prioritization
  • A vulnerability with evidence of active impact in logs ranks higher than one with no log activity
8DVulnerability Analysis

Vulnerability Analysis — Prioritization & Classification

Vulnerability analysis transforms raw scan findings into a prioritized remediation list. It answers the question every security team faces after a scan: we found hundreds of vulnerabilities — where do we start?

🏆
Prioritization — What to Fix First
Ease of exploitation — is there a public exploit available? Does it require authentication? Remote or local?
Potential damage — what is the worst-case outcome? Data loss, system compromise, lateral movement?
Asset value — critical production database vs. isolated test system? Does the asset hold regulated data?
Threat landscape — is this being actively exploited in the wild? Is it referenced in current threat feeds?
📂
Classification — Organizing by Type
Categorizing vulnerabilities by: type of system affected (OS, web app, network device, database), nature of the vulnerability (injection, authentication weakness, cryptographic flaw, misconfiguration), and potential impact category (CIA Triad dimension affected). Classification routes findings to the right remediation teams and reveals patterns — many injection flaws suggest a training or code review gap.
📉
Exposure Factor (EF)
Measures the extent to which an asset is susceptible to being compromised. High exposure factors: weak authentication, inadequate network segmentation, insufficient access controls, internet accessibility, and sensitive data stored on the affected system. The more accessible the system and the more valuable the data — the higher the exposure factor and the higher the remediation priority.
🌍
Environmental Variables
IT infrastructure — legacy systems, new technologies, complexity. External threat landscape — active ransomware campaigns in your sector. Regulatory environment — healthcare and finance face specific compliance deadlines. Operational environment — poor patch management, weak access controls, or insufficient security training all increase exposure to specific vulnerability types.
⚖️
Risk Tolerance Alignment
Vulnerability analysis must align with the organization's risk tolerance — the level of risk it is willing to accept. Low risk tolerance: aggressive remediation timelines for all severity levels. Higher risk tolerance: formally accepts certain low-severity vulnerabilities on non-critical systems. Risk acceptance must be formally documented, management-approved, and periodically reviewed — not made unilaterally by technical staff.
8DResponse & Remediation

Vulnerability Response, Remediation & Validation

P
Patching — Most Direct Remediation
Applies vendor-released updates to fix known vulnerabilities at their source. Covers: operating systems, network devices, databases, web applications, desktop applications. Requires centralized, automated patch management — applied promptly and consistently. Testing before deployment is mandatory — a patch that breaks production creates its own incident.
S
Segmentation — Contain the Blast Radius
Network segmentation isolates a vulnerable system so that if it is compromised, the attacker is confined to that segment. Prevents lateral movement across the entire network. Used as both a remediation technique and a compensating control when patching is not immediately possible.
CC
Compensating Controls — When Direct Remediation Waits
Measures put in place when direct remediation is not immediately possible. Examples: additional monitoring around the vulnerable system, secondary authentication requirements, enhanced encryption for data the vulnerability might expose, network restrictions limiting which systems can communicate with the vulnerable host. Compensating controls reduce risk — they do not eliminate the vulnerability.
EX
Exceptions and Exemptions — Formal Risk Acceptance
When a vulnerability cannot be remediated due to business criticality, technical constraints, or cost. Requirements: senior leadership formally accepts the risk (not technical staff alone), rationale is documented, a reassessment timeline is established, and the exception is reviewed periodically. Risk acceptance decisions do not last forever.
V
Validation — Confirming the Fix Worked
Re-scanning: run the same scan after remediation — if the finding no longer appears, remediation was successful. Auditing: review the steps taken and confirm documentation is updated. Verification: manual checks, automated testing, or system log review confirming the fix functions correctly and did not introduce new issues.

Why validation matters: a patch applied to 19 out of 20 affected systems still leaves one exposed. Remediation without validation is an assumption — not a conclusion.
📊 Vulnerability Reporting — What Reports Must Include
  • Vulnerability inventory with CVSS ratings — all findings ranked by severity and potential impact
  • Impact descriptions — what could happen if each vulnerability is exploited, in business language management understands
  • Remediation recommendations — specific patches, configuration changes, or compensating controls
  • Timeliness — every day between finding and reporting extends the attacker's window of opportunity
  • Dual audience clarity — technical staff need specific details; management needs risk context and business impact

What You Have Learned

🔄

VM Lifecycle

Identify → Evaluate → Treat → Report → Repeat. Vulnerability management is continuous — not a one-time project. New vulnerabilities emerge daily.

🖥️

OS Vulnerabilities

Windows (MS17-010/WannaCry), macOS (Shellshock), Linux (Heartbleed), Android (Stagefright), iOS (watering hole). Each platform has a vulnerability history.

🔒

Vulnerability Types

Legacy, EOL, firmware (Meltdown/Spectre, LoJax), virtualization (VM escape, resource reuse, hypervisor). Know real-world examples for each.

⚡

Zero-Day

Previously unknown — no signature, no patch, no traditional defense. Worth millions. Used by nation-states and organized crime. Responsible disclosure limits harm.

⚙️

Misconfiguration

Default credentials, unnecessary services, cloud storage left public. Support activities introduce risk. Change management discipline prevents it.

🔐

Cryptographic Vulns

MD5/SHA-1 broken (collisions). DES/3DES deprecated (Sweet32). AES is current standard. BEAST, POODLE, Heartbleed, KRACK are real-world examples.

📱

Mobile Vulnerabilities

Rooting (Android), jailbreaking (iOS), sideloading — all bypass security controls. MDM platforms detect and restrict these practices. Excessive permissions increase attack surface.

🌐

Web App Attacks

XSS (reflected, stored, DOM-based), SQL injection, CSRF. All exploit insufficient input validation and session management. Root cause: insecure coding practices.

☁️

Cloud Attacks & CASB

Side-channel attacks, cryptojacking, cloud as attack platform. CASB: forward proxy, reverse proxy, API-based modes. Mediates all cloud service access.

📦

Supply Chain & SBOM

Service providers, hardware suppliers, software providers all introduce risk. SBOM = nutrition label for software. OWASP Dependency-Check, CycloneDX, SPDX.

🔍

Scanning & Threat Intelligence

Credentialed vs. non-credentialed. CVE/NVD/SCAP/CVSS. Threat feeds, ISACs, OSINT tools. Dark web as threat intelligence source. Pen testing: unknown/known/partial.

🛠️

Analysis & Remediation

Prioritize by exploitability + impact + asset value + threat landscape. Patch, segment, compensate, or accept. Validate with re-scanning, auditing, and verification.

Key Terminology

Lesson 8 Glossary

Essential terms and definitions for the CompTIA Security+ SY0-701 exam.

ATA Secure Erase
SSD-specific command instructing the drive's firmware to sanitize all stored data, including inaccessible marked-as-bad memory cells. Traditional overwriting does not work on SSDs.
BEAST Attack
Browser Exploit Against SSL/TLS — exploited cipher suite implementation weaknesses in SSL and early TLS. One of the attacks that drove adoption of TLS 1.2 and 1.3.
CASB
Cloud Access Security Broker — enterprise software mediating access to cloud services. Implements SSO, malware scanning, activity monitoring, and data exfiltration prevention via forward proxy, reverse proxy, or API modes.
CVE
Common Vulnerabilities and Exposures — standardized dictionary of vulnerabilities. Format: CVE-YYYY-####. Maintained by MITRE at cve.mitre.org. Provides universal reference point.
CVSS
Common Vulnerability Scoring System — generates scores 0–10 based on vulnerability characteristics. Bands: None (0), Low (0.1–3.9), Medium (4.0–6.9), High (7.0–8.9), Critical (9.0–10.0). Maintained by FIRST.
CycloneDX
Lightweight SBOM specification designed for streamlined security-focused SBOM data sharing and analysis. OWASP standard increasingly adopted for vulnerability management workflows.
Cryptojacking
Attacker uses victim's cloud computing resources to mine cryptocurrency without consent — resulting in vastly increased cloud costs and degraded performance for the victim.
DOM-Based XSS
Cross-site scripting attack targeting client-side DOM scripts. The server never sees the malicious payload — it executes entirely in the victim's browser. Difficult to detect server-side.
EternalBlue
NSA-developed exploit leveraging MS17-010 SMB protocol vulnerabilities. Leaked and used by WannaCry ransomware in May 2017, causing global damage. Demonstrates stockpile risk.
Exposure Factor (EF)
Measures the extent to which an asset is susceptible to being compromised by a specific vulnerability. Higher accessibility and higher data sensitivity = higher exposure factor.
False Negative
A real vulnerability that exists but is not detected by a vulnerability scan. Creates a false sense of security. Mitigated by multi-vendor scanning, repeat scans, and penetration testing.
False Positive
A vulnerability scan incorrectly identifies a vulnerability that does not actually exist. Excessive false positives cause alert fatigue — analysts begin ignoring scan results entirely.
Heartbleed
CVE-2014-0160 — critical flaw in OpenSSL allowing attackers to read server memory contents including private keys and credentials. Affected the majority of Linux servers worldwide in 2014.
Jailbreaking
Removing Apple's iOS restrictions — enabling unauthorized apps, system file access, and bypassing security controls. Violates Apple's terms of service — permanently removes manufacturer support and patches.
KRACK
Key Reinstallation Attacks — vulnerability in the WPA2 protocol protecting Wi-Fi traffic. Allowed attackers within range to intercept and decrypt sensitive network traffic.
LoJax
2018 UEFI firmware vulnerability enabling malware persistence even after complete hard drive replacement or OS reinstallation. Cannot be removed by standard remediation methods.
Meltdown & Spectre
2018 processor-level vulnerabilities affecting virtually all computers and mobile devices. Allowed malicious programs to steal data as it was being processed. Required hardware-level mitigation.
MS17-010 / EternalBlue
Critical SMB protocol vulnerability exploited by WannaCry ransomware in May 2017. Systems that had applied the March 2017 patch were protected. Systems that had not were compromised at scale globally.
NVD
National Vulnerability Database — maintained by NIST at nvd.nist.gov. Supplements CVE entries with CVSS severity ratings, affected software versions, and mitigation measures. The authoritative vulnerability reference.
OSINT
Open-Source Intelligence — collecting and analyzing publicly available information. Tools: Shodan, Maltego, Recon-ng, theHarvester. Adds real-world exploitation context to CVSS-based prioritization.
POODLE
Padding Oracle On Downgraded Legacy Encryption — forced browsers to downgrade to SSL 3.0, then exploited its known weaknesses. Demonstrated the risk of supporting legacy cipher suite versions.
Responsible Disclosure
The process by which ethical researchers privately notify vendors of a zero-day before public disclosure — giving time to develop a patch. Limits potential harm from discovery.
Rooting
Gaining root (administrative) access on an Android device. Bypasses security boundaries — malicious apps can access system files and corporate data. MDM controls may be bypassed.
SBOM
Software Bill of Materials — comprehensive inventory of all components in a software product including dependencies, libraries, frameworks, component versions, and supplier information. The "nutrition label" for software.
SCAP
Security Content Automation Protocol — NIST framework for automating vulnerability management, distributing feed updates, and comparing system configurations against secure baselines. Provides common identifiers across tools.
Shellshock
2014 vulnerability in the Bash shell affecting all UNIX-based systems including macOS. Allowed arbitrary command execution with no special privileges required — exploitable remotely via web servers running CGI scripts.
Sideloading
Installing applications outside official app stores (Google Play, Apple App Store). Android: APK files from third-party sources. iOS: typically requires jailbreaking. Bypasses the vetting process — enables malicious app installation.
Side-Channel Attack
Cloud-specific attack where an attacker provisions an instance on the same physical server as the victim — extracting information through observation of shared physical resources (CPU cache, memory).
SQL Injection (SQLi)
Adding malicious SQL code to input accepted by a web application — causing the database to execute the attacker's query instead of the intended one. Root cause: insufficient input sanitization.
Stagefright
2015 Android vulnerability in the Stagefright media library — present on virtually all Android devices. Attackers could execute malicious code by sending a specially crafted MMS message with no user interaction required.
Sweet32
CVE-2016-2183 — birthday attack against 3DES exploiting its 64-bit block size. Published August 2016. Led NIST to officially deprecate 3DES in 2017 with recommended discontinuation by 2023.
VM Escape
Attack where a threat actor breaks out of a virtual machine's isolated environment and gains access to the host system or other VMs. Cloudburst (CVE-2009-1244) is the defining example.
XSS
Cross-Site Scripting — exploits browser trust in scripts from sites the user visits. Three types: Reflected (crafted link), Stored/Persistent (database injection), DOM-Based (client-side script manipulation). Root cause: insufficient input validation.
Zero-Day Vulnerability
A previously unknown software or hardware flaw that attackers can exploit before developers are aware of it. No patch exists — traditional signature-based defenses are ineffective. Can be worth millions of dollars on the exploit market.
