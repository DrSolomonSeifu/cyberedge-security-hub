Lesson 5 — Introduction

Overview and Learning Objectives

Securing a network is not only about controlling who logs in. It is also about how the network is physically and logically designed. This lesson examines how thoughtful network design — combined with the right security controls — supports the CIA Triad at every layer.

🎯 Lesson Objectives

  • 01 Compare and contrast the security implications of different on-premises network architecture models — identifying what makes one design more secure than another.
  • 02 Apply security principles — including least privilege, segmentation, and defense in depth — to evaluate and strengthen on-premises network architecture.
  • 03 Select effective controls — choosing the right security appliances and placing them in the correct locations within the network topology.
  • 04 Ensure secure communications for remote access and tunneling — implementing VPN protocols and secure remote access tools so off-site users can connect safely.
💡 Why This Matters You may not design networks on day one — but you will work inside them every day. Understanding how networks are built is what allows you to spot weaknesses, contribute to planning, and make sound security recommendations.
Three Guiding Principles

Foundations Behind Every Decision

Every architecture and control decision in this lesson connects back to these three principles.

🔒

CIA Triad

All secure network design must support Confidentiality (keep data private), Integrity (keep data accurate), and Availability (keep systems running). Every design decision protects one or more of these properties.

🛡️

Defense in Depth

No single control is enough. Layer multiple controls across multiple levels — so if one fails, others remain. Think of it as multiple locks rather than just one.

⚖️

Least Privilege

Every user, device, and network zone gets only the minimum access needed to do its job — nothing more. All other traffic is blocked by default, limiting damage if any part is compromised.

Sections 5.1.1 – 5.1.4

Network Infrastructure

Before securing a network, you must understand how it is physically and logically built — from the cables in the walls to the protocols that route data between zones.

📐 Network Architecture — Three Layers

Network architecture refers to the selection and placement of media, devices, protocols, and data assets. It operates across three layers:

  • Network Infrastructure — the physical foundation: cabling, switches, routers, and basic connectivity protocols. This is the "road system" data travels on.
    • Includes the cables, appliances, and addressing protocols that keep devices connected.
  • Network Applications — services running on the infrastructure to support business activity, such as email, file sharing, and web hosting.
  • Data Assets — the actual information the business creates, stores, and transfers, including customer records, financial data, and emails.
Key principle: Secure architecture exists to support secure business workflows. Every design decision must preserve Confidentiality, Integrity, and Availability.

🔢 The OSI Model — Key Layers for Security

The OSI model breaks a network into distinct layers. Security controls operate at specific layers — knowing which layer a problem lives at helps you diagnose and fix it faster.

LayerNameDevice / ProtocolIdentifier Used
Layer 1PhysicalCable, fiber, wirelessNo addressing — raw signals
Layer 2Data LinkSwitch, Access PointMAC address (hardware ID)
Layer 3NetworkRouterIP address and subnet
Layer 4TransportTCP / UDPPort numbers
Layer 7ApplicationHTTP, SMTP, DNS, FTPFQDN (domain name)

🔀 Switches — Layer 2 Forwarding

Switches connect devices within a local network and forward data frames using MAC addresses — the unique hardware identifier on every network adapter.

  • Structured cabling path: PC → patch cable → wall port → conduit → patch panel → switch port. This standard layout is used in offices and campus buildings.
  • Star topology — the switch sits at the center; all devices connect outward from it. The switch can link any two connected hosts.
  • Broadcast domain — all hosts on the same switch share a broadcast domain, meaning they all receive broadcast messages from one another.
  • Security risk — flat network: Any host can freely attempt to reach any other host on the same segment. A single compromised device can attack everything else.
    • This is why large networks divide the network into segments rather than using one large flat switch.

🏗️ Hierarchical Network Design

To overcome the risks of a flat network, enterprise networks use a hierarchical design that divides the network into separate, manageable blocks.

  • Access blocks — the network is divided into groups of hosts, each served by its own access switch. Each block is a star topology for that group of devices.
  • Core router or Layer 3 switch — connects all access blocks at the center. Controls and routes traffic between blocks, creating separate broadcast domains.
    • A Layer 3 switch combines switching and routing functions in a single appliance.
  • Zone-based security model — because each block can have its own access policies, this design allows different rules for different parts of the network.
    • Example: workstations in one block, servers in another, printers in a third, guest Wi-Fi completely isolated.
  • Server rooms — servers and core networking equipment are placed in secure, dedicated rooms and connected directly to switch ports via patch cables.
⚠️ Remember: A flat internal network gives an attacker complete freedom of movement once they breach the perimeter. Hierarchical design and segmentation are not optional — they are essential.

🌐 Routing and IP Addressing

Routers forward traffic between network segments using IP addresses. Each subnet becomes its own broadcast domain — traffic between subnets must pass through the router, where access rules are applied.

  • IPv4 — uses dotted decimal notation with a prefix. Example: 10.1.32.0/24 means the first 24 bits identify the network; the remaining bits identify individual hosts.
    • This /24 prefix can also be written as subnet mask: 255.255.255.0
  • IPv6 — uses 128-bit addresses in hexadecimal format. Has a built-in address hierarchy for global ISP routing and local subnetting.
  • ARP (IPv4) and Neighbor Discovery (IPv6) — map IP addresses to MAC addresses so data can be delivered across the physical network.
  • Each access block gets its own subnet — making it straightforward to write precise access control rules between zones. Example: deny all traffic from 192.168.42.0/24 (guest) to 10.1.24.0/24 (database servers).

📡 VLANs — Virtual Local Area Networks

A VLAN allows you to logically separate devices even when they are on the same physical switch — eliminating the need to buy separate hardware for every network segment.

  • How VLANs work — each VLAN is assigned a unique ID between 2 and 4,094. Any switch port can belong to any VLAN, regardless of physical location on the network.
  • Traffic isolation — each VLAN is its own Layer 2 domain. Devices on different VLANs cannot communicate directly; their traffic must pass through a router or Layer 3 switch first.
    • Example: A VoIP handset on VLAN40 and a workstation on VLAN32 share the same physical switch — but cannot reach each other without going through the router.
  • Access control at the router — when traffic between VLANs passes through the router, access control rules determine whether to allow or deny it. This is where security policy is enforced.
  • Scalability — VLANs extend across multiple switches. Adding a new floor simply requires configuring the same VLAN IDs on the new switch — no redesign needed.

🔌 Port Security

Every wall port is a potential entry point. An attacker with physical access can plug in a device and join the network. Three levels of control address this risk:

  • Physical controls — place switches in locked rooms, disable unused switch ports, or physically remove patch cables from unused wall ports.
    • Limitation: an attacker can still unplug an active device and connect their own machine to an enabled port.
  • MAC filtering / MAC limiting — configure a switch port to allow only specific MAC addresses, or limit the number of allowed MACs.
    • Weakness: MAC addresses can be spoofed (faked), and managing large MAC address lists is difficult at scale.
  • IEEE 802.1X (PNAC) — requires full authentication before the port grants any network access. Uses EAP (authentication protocol) and RADIUS (authentication server). The enterprise-grade standard.
    • The port stays closed until the connecting device successfully authenticates — providing the strongest control.

🌬️ Physical Isolation — Air Gaps

Some systems are so sensitive that connecting them to any network is unacceptable. The solution is complete physical separation — called an air gap.

  • Air-gapped host — a single machine with no cabled or wireless connection to any other network. Used for: root Certificate Authority servers, malware analysis machines.
  • Air-gapped network — hosts within the network can communicate with each other, but there is no external connection. Used in military, government, and industrial facilities.
  • Management tradeoff — all administration must happen locally; updates must be delivered via USB or optical media.
    • That media becomes a potential attack vector and must be scanned before use — as demonstrated in real-world industrial attacks.

⚙️ Architecture Considerations

When evaluating any architecture or control, weigh these factors against each other:

  • Cost — up-front capital costs plus ongoing maintenance. Weigh cost against the losses a breach would cause.
  • Scalability — can the system expand when workloads increase without a complete redesign? On-premises networks typically have low scalability.
  • Availability and Resilience — minimize downtime; ensure the system can recover quickly from failures without manual intervention.
  • Patch availability — unsupported or end-of-life devices cannot receive security patches and remain permanently vulnerable.
  • Risk transference — a third-party contract (SLA) can shift responsibility for uptime and performance, but does not eliminate the underlying risk.
Sections 5.1.5 – 5.1.6

Security Zones and Attack Surface

Segmentation is how we control the blast radius of any security incident. Zones assign different trust levels to different parts of the network — and traffic between them is strictly controlled.

🗺️ Zone-Based Security Model

A security zone is a group of hosts with similar trust levels and access requirements. Traffic between zones must pass through a firewall that enforces least privilege — blocking everything not explicitly permitted.

ZoneExamplesDefault Access Rule
High PrivilegePrivate app servers, database serversAccepts limited inbound from medium — never initiates outbound
Medium PrivilegeWorkstations, VoIP handsetsCan make requests outward — cannot accept new inbound connections
Low PrivilegePrintersAccepts inbound connections (e.g., print jobs) — cannot initiate outbound
UntrustedGuest Wi-Fi, public-facing serversInternet access permitted — completely blocked from the internal network
InternetEverything beyond the border routerFully untrusted — all access governed by the border firewall
Important rule: Every zone must have a known and controlled entry and exit point. Adding an unauthorized wireless access point inside a zone destroys its security — it creates an uncontrolled entry that bypasses all existing controls.

🎯 The Network Attack Surface

The attack surface is every point where a threat actor could gain unauthorized access. It exists at every OSI layer — both externally (from the internet) and internally (from inside the network).

  • Layer 1/2 — Physical access: An attacker plugs a device into a wall port or joins an open wireless network, gaining access to the broadcast domain.
    • Controls: port security, 802.1X authentication, wireless authentication.
  • Layer 3 — IP spoofing: An attacker fakes a valid IP address to impersonate a trusted device and communicate across zone boundaries.
  • Layer 4/7 — Service exploitation: An attacker connects to an open TCP or UDP port and interacts with an application or service — where most real-world database, web, and email attacks occur.

Defense in depth means deploying multiple control types — preventive, detective, and corrective — across all these layers simultaneously.

⚠️ Common Network Architecture Weaknesses

These are vulnerabilities analysts encounter regularly. Knowing them is the first step to fixing them.

  • Single points of failure — one device or link that everything depends on. If it fails, the whole network fails. Resilient design requires redundancy.
  • Overdependence on perimeter security — if the internal network is flat, one breach gives an attacker complete freedom of movement. Internal segmentation is not optional.
  • Availability over confidentiality and integrity — taking shortcuts to bring a service online quickly creates long-term vulnerabilities. Security should never be bypassed for speed.
  • Lack of documentation and change control — undocumented devices and changes create unknown gaps. You cannot protect what you cannot see.
  • Complex dependencies — services that rely on many other systems simultaneously. A single failure should not cascade into a total outage.
Section 5.2 — Network Controls

Network Security Appliances

Security appliances are the devices that enforce access control policies between and within zones. Selecting the right appliance and placing it correctly are equally important.

📍 Device Placement — Three Control Types

Every security appliance is placed with a specific role in mind. The three types form the layers of defense in depth:

  • Preventive controls — placed at zone borders. They stop threats before they enter. Examples: firewalls (enforce access rules), load balancers (prevent single-server overload).
  • Detective controls — placed inside the perimeter, monitoring internal traffic. They catch what prevention missed. Example: IDS sensor attached to a mirror port behind the firewall.
  • Corrective controls — placed inline to respond actively to detected threats. Example: IPS that blocks an attacker's IP address automatically upon detection.

All three types can also be installed directly on endpoints (host firewall, antivirus, DLP) as a final layer of protection if network controls are bypassed.

⚙️ Active vs. Passive Controls

Passive Controls

  • Receive a copy of traffic — do not sit in the live path
  • Hosts are completely unaware of their operation
  • Have no IP address — cannot be targeted or attacked
  • Example: IDS sensor on a mirror port

Active Controls

  • Sit in the live traffic path — interact directly
  • Can block, modify, or redirect traffic in real time
  • Hosts may need to be configured to use them
  • Example: IPS deployed inline; forward proxy

📡 TAP vs. Mirror Port (SPAN)

An IDS needs a copy of network traffic to analyze. Two methods deliver that copy — and they are not equally reliable:

TAP (Test Access Point)

  • Hardware device placed inline on the cable itself
  • Physically copies every signal — including corrupted frames
  • Never drops frames, even under maximum network load
  • Most reliable — preferred for critical network links

Mirror Port (SPAN)

  • Software feature configured on a switch
  • Does not copy frames containing errors
  • May drop frames when the switch is under heavy load
  • Easier to deploy but less complete and reliable

⚡ Fail-Open vs. Fail-Closed

When a security device fails — from power loss, hardware fault, or software bug — its behavior determines whether security is maintained:

  • Fail-open — access is preserved when the device fails. Prioritizes Availability.
    • Risk: an attacker could deliberately trigger a failure to bypass the control entirely.
  • Fail-closed — all access is blocked when the device fails. Prioritizes Confidentiality and Integrity.
    • Risk: the network stops working until the device is restored. The security boundary is preserved.
Rule of thumb: Border firewalls should almost always fail-closed — the security boundary must hold even in failure. Internal monitoring sensors may be acceptable to fail-open, since other controls remain active.

🔥 Firewalls — Layers 3, 4, and 7

A firewall is the primary preventive control — it enforces access policies using an Access Control List (ACL): a set of rules defining what traffic is permitted, dropped, or rejected. Three action types:

  • Accept / Permit — the packet passes through.
  • Drop / Deny — the packet is silently discarded. The sender receives no response.
  • Reject — the packet is blocked, and the sender receives an ICMP "port unreachable" message.

Firewalls inspect traffic at increasing depth:

  • Stateless packet filtering — inspects each packet independently using IP address, protocol type, and port number. Fast but cannot detect attacks spread across multiple packets.
  • Stateful inspection (Layer 4) — tracks active sessions in a state table. Verifies each packet belongs to an established, legitimate session. Detects SYN floods and session hijacking.
  • Deep packet inspection (Layer 7) — inspects the actual content of packets. Verifies that application protocols match their expected ports and can detect malicious code strings.

Firewalls are deployed in three placement modes:

  • Routed (Layer 3) — has IP and MAC addresses; routes between subnets. Most common for zone border firewalls.
  • Bridged (Layer 2) — has MAC address but no IP; inspects traffic passing between two nodes like a switch.
  • Inline / Transparent (Layer 1) — no addresses at all; acts like a cable segment (bump-in-the-wire). Deployed without reconfiguring any surrounding devices.

🔄 Proxy Servers

A proxy uses a store-and-forward model — it fully deconstructs each packet, inspects it, rebuilds it, and only forwards it if it passes the rules. This is deeper inspection than a basic firewall.

  • Forward proxy — controls outbound traffic from internal users to the internet. Can cache frequently visited pages (improving performance), apply content filtering, and block access to banned sites or IP addresses.
    • Non-transparent: clients must be configured with the proxy address (typically port TCP/8080).
    • Transparent: intercepts traffic automatically — no client configuration needed. Implemented as an inline appliance.
  • Reverse proxy — controls inbound traffic from the internet toward internal servers. External users only ever contact the reverse proxy — never the internal server directly. Protects servers from direct exposure.

🔍 IDS and IPS

Where a firewall uses rules to filter traffic, IDS and IPS look for signs of attack within the traffic that has already passed through. Common platforms: Snort, Suricata, Zeek.

  • IDS — Intrusion Detection System — passive detective control. Receives copies of traffic via TAP or mirror port; matches against known signatures and behavioral patterns. Raises alerts but does NOT block anything. Undetectable because it has no addressable interface.
  • IPS — Intrusion Prevention System — active corrective control. Responds automatically when an attack is detected:
    • Shunning — blocks the source IP address temporarily or permanently.
    • Connection reset — terminates the session without revealing detection.
    • Redirect to honeypot / honeynet — diverts the attacker to a decoy system for further analysis and intelligence gathering.

🚀 NGFW and UTM

  • NGFW (Next-Generation Firewall) — combines Layer 7 deep packet inspection, IPS functionality, TLS traffic decryption, and identity-aware filtering (per-user or per-role rules) in one appliance. The modern enterprise standard since 2010.
  • UTM (Unified Threat Management) — centralizes firewall, antimalware, IPS, spam filtering, content filtering, DLP, VPN, and endpoint protection into a single managed console.
    • Advantage: simplified management — one console for everything.
    • Risk: creates a single point of failure; can struggle with latency under high network load; may underperform dedicated single-function appliances.

⚖️ Load Balancers

A load balancer distributes incoming client requests across multiple servers in a farm, directly supporting Availability in the CIA Triad. It also mitigates denial of service attacks.

  • Layer 4 load balancer — forwards based on IP address and port number. Fast and simple.
  • Layer 7 load balancer (content switch) — forwards based on application content such as URL paths or data types. Used for complex modern web applications.
  • Scheduling algorithms — determine which server handles each request:
    • Round robin — cycles through servers in order. Simplest method.
    • Fewest connections — sends to the least busy server.
    • Best response time — sends to the fastest responding server.
  • Session persistence — keeps a client connected to the same server throughout a session:
    • Source IP affinity (Layer 4) — client is routed to the same server based on IP address.
    • Cookie-based persistence (Layer 7) — more reliable; uses a browser cookie to track and maintain the session.

🌐 Web Application Firewall (WAF)

A WAF specifically protects web servers and their databases from code injection attacks (SQL injection, cross-site scripting) and denial of service attacks, operating at Layer 7.

  • Uses a signature database and pattern matching to identify malicious strings within HTTP request content — not just headers and ports.
  • All detected events are logged — providing evidence for investigation and incident response.
  • Deployment options:
    • Standalone appliance — protects the entire web server zone.
    • Plug-in software — installed directly on the web server (e.g., ModSecurity on IIS or Apache).
Topic 5C — Remote Access and VPN

Secure Communications

Most organizations need to support connections from employees, contractors, and customers working outside the trusted network. This topic covers how to secure those connections — even when they travel across the public internet.

🔑 What is a VPN?

A Virtual Private Network (VPN) creates an encrypted tunnel between a remote device and a private network over an untrusted public network. Even if packets are intercepted in transit, the contents remain unreadable.

A VPN does not make an untrusted network trusted — it makes communication over that network private. Authentication, encryption, and access controls must all be enforced separately.

⚠️ Deprecated: PPTP (Point-to-Point Tunneling Protocol) no longer meets modern security standards. Always use TLS or IPsec instead.

👤 Remote Access VPN — Client-to-Site

Used when individual remote users (home workers, field staff) need to connect to the corporate network. Also called the telecommuter model.

  • The user's device (VPN client) connects to a VPN gateway at the network edge using any internet connection — home broadband, mobile data, hotel Wi-Fi.
  • The gateway authenticates the user and creates an encrypted tunnel. Traffic is then routed into the internal network.
  • VPN access does not mean access to everything — access control policies still apply after the tunnel is established.
  • The VPN gateway typically sits in a screened subnet — a controlled buffer zone — with a firewall on both sides to inspect inbound and outbound traffic.

🏢 Site-to-Site VPN

Used to connect two entire office networks permanently — for example, a branch office to a head office. Unlike remote access VPN, the connection is always-on and automatic.

  • Both sites have their own VPN gateway. Gateways exchange security credentials and maintain the tunnel automatically — no user action required.
  • Individual hosts at each site do not know the VPN exists. The routing infrastructure handles traffic automatically: local traffic stays local; cross-site traffic goes through the tunnel.
  • The tunnel terminates at the head office screened subnet, with a router and firewall controlling access to the internal LAN.

🔐 TLS VPN — Transport Layer Security

TLS VPN uses digital certificates to secure the connection — the same technology behind HTTPS. It operates at the application layer and is commonly used for remote access VPN.

  • Server authentication — the VPN gateway presents its certificate, proving it is the legitimate server and not an impostor. Always required.
  • Mutual authentication — optionally, the client also presents its certificate. Both sides verify each other's identity before the tunnel is fully established. Significantly stronger than one-way authentication.
  • User authentication — inside the encrypted tunnel, the user submits credentials (username/password) verified by a RADIUS server on the internal network.
  • Protocol and port — can operate over UDP (preferred — lower overhead) or TCP (fallback). OpenVPN default: 1194/UDP.

🛡️ IPsec VPN — Internet Protocol Security

IPsec operates at Layer 3, securing every IP packet transmitted. It is the standard for site-to-site VPN deployments. Two protocols provide different levels of protection:

AH — Authentication Header

  • Signs the packet — proves its origin and detects tampering
  • Does NOT encrypt — payload is readable
  • Provides integrity only — rarely used for VPN tunnels

ESP — Encapsulating Security Payload

  • Encrypts the payload — hides content from interception
  • Also provides authentication and integrity
  • The real-world standard for all VPN deployments

IPsec operates in two modes:

  • Transport mode — used between two hosts on a private network. The payload is encrypted; the original IP header remains visible. Used for host-to-host communication, not typically for VPN tunnels.
  • Tunnel mode — used between VPN gateways across the internet. The entire original packet — header and payload — is encrypted and wrapped inside a new IP packet. The original addresses are completely hidden. Standard for site-to-site VPN.

🔑 IKE — Internet Key Exchange

Before two IPsec peers can communicate, they must agree on encryption settings. IKE handles this negotiation and establishes the Security Association (SA) — the agreed-upon cryptographic parameters.

  • Phase 1 — Authentication — both peers prove their identity using either:
    • PKI digital certificates (stronger — each peer is issued a certificate by a mutually trusted CA).
    • Pre-shared key (simpler — same passphrase configured on both peers, but less secure).
  • Phase 2 — Cipher negotiation — both peers agree on encryption algorithms, key sizes, and whether AH or ESP will be used for the actual data transfer.
  • IKEv1 — supports host-to-host and site-to-site topologies only.
  • IKEv2 — adds EAP support for RADIUS-based user authentication, NAT traversal for home routers, and MOBIKE for mobile clients that switch between Wi-Fi and cellular without dropping the VPN connection.

🖥️ Remote Desktop

Remote desktop tools allow a user to see and control a specific remote computer as if seated in front of it — transmitting screen, audio, keyboard, and mouse data over the network.

  • RDP (Remote Desktop Protocol) — Microsoft's standard. One-to-one access to a physical machine. Encrypted by default. A remote desktop gateway can also provide access to virtual desktops or individual applications hosted on servers.
  • Alternatives — TeamViewer and VNC (Virtual Network Computing) support cross-platform access including Windows, macOS, Linux, and mobile devices.
  • HTML5 / Clientless gateway — remote desktop access directly through a web browser with no software installation. Uses the HTML5 canvas element and the WebSocket protocol for real-time bidirectional communication. Example: Apache Guacamole.

💻 Secure Shell (SSH)

SSH is the standard protocol for secure command-line remote administration — used to manage servers, routers, switches, and firewalls. Also used for secure file transfer via SFTP and SCP.

  • Host key — every SSH server has a public/private key pair that uniquely identifies it. On first connection, the client is shown a fingerprint of the server's key. Always verify this fingerprint with the server administrator through a separate trusted channel before accepting — an unverified key risks a man-in-the-middle attack.
  • Client authentication methods:
    • Username/password — simplest method. Credentials verified by local database or RADIUS server.
    • Public key authentication — client's public key is pre-installed on the server. Stronger than password — the private key is never transmitted.
    • Kerberos — single sign-on via domain controller. Client submits a Kerberos ticket obtained at workstation login.
  • ssh vs. scp — ssh opens an interactive terminal session for administration; scp (Secure Copy) transfers files securely over the same encrypted channel without an interactive session.

🖧 Out-of-Band Management and Jump Servers

Out-of-band (OOB) management uses a network path that is completely separate from production traffic — so administrators can still reach and manage devices even when the main network is down or compromised.

  • OOB methods:
    • Serial or console port — direct physical connection to the device's management port.
    • Virtual terminal — management interface accessible over a dedicated management VLAN or separate physical network.
  • SAW (Secure Administrative Workstation) — a dedicated, locked-down machine used exclusively for remote administration. Only essential software installed; internet access denied or restricted to approved vendor sites.
  • Jump server — a single hardened server inside a secure zone running only SSH or RDP. Administrators connect to the jump server first — never directly to application servers.
    • Each application server's ACL contains only the jump server's IP address. All other connection attempts are denied.
    • Creates a single, controlled, auditable entry point — greatly simplifying monitoring and access management.
    • Remote admins connect via VPN to reach the jump server. Internal admins connect from a SAW. Unauthorized workstations are blocked.
