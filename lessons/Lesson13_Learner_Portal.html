Lesson Overview

Learning to read the evidence

The previous lesson gave you the tools and data sources that surface suspicious activity. This lesson is the analyst's core skill: taking that raw evidence and recognizing it for what it is — an indicator of malicious activity.

An indicator rarely arrives labeled. A single odd log line could be nothing. The analyst's job is to know the patterns each kind of attack leaves behind, and to correlate weak signals into a confident finding. This lesson walks through three families of attack and, for each, the indicators that give it away.

🔑 Key Idea
An Indicator of Compromise (IoC) is observable evidence that an attack has occurred or is underway. No single indicator is proof — analysts build confidence by correlating many of them. Knowing the expected indicators for each attack type is what turns raw data into detection.

Three Indicator Categories

Whatever the attack, its indicators tend to fall into three buckets. Keeping these in mind helps you organize what you see.

🌐

Network Indicators

Traffic spikes, connections to suspicious addresses, unusual ports, beaconing, large transfers.

💻

Host Indicators

New processes, unauthorized startup items, file and registry changes, malware artifacts, missing logs.

👤

Behavioral Indicators

Impossible travel, concurrent sessions, off-hours activity, privilege use that does not fit the account.

Topic 13A · Malware Attack Indicators

Classifying hostile code and the traces it leaves

📖 Definition
Malware is any software written to cause harm or give an attacker unauthorized control. Types are classified two ways: by how they spread and by what they do once active.

The Malware Families

🧬

Virus

Attaches to a file or program and spreads when that host is run by a user.

Tell: corrupted or modified files
🪱

Worm

Self-replicating; spreads across networks on its own, with no user action needed.

Tell: rapid network spread, bandwidth spikes
👻

Fileless Malware

Runs in memory using built-in tools like PowerShell, leaving little on disk.

Tell: suspicious script/process activity
🐴

Trojan

Disguised as something legitimate to trick the user into running it.

Tell: unexpected behavior from a "safe" app
🚪

Backdoor / RAT

Gives an attacker remote control of the host and persistent re-entry.

Tell: outbound C2 connections
🕵️

Spyware / Keylogger

Secretly collects activity, keystrokes, or credentials. Adware injects unwanted ads.

Tell: stolen credentials, odd processes
🥷

Rootkit

Hides deep in the OS (even kernel level) to conceal other malware from tools.

Tell: tools report inconsistent results
🔒

Ransomware

Encrypts files and demands payment. Related: cryptojacking (steals compute to mine) and logic bombs (trigger on a condition).

Tell: mass rapid file encryption

TTPs and IoCs

🔎 Two Lenses on an Attack
TTPs (Tactics, Techniques, and Procedures) describe how a threat actor operates — their playbook. IoCs (Indicators of Compromise) are the concrete artifacts that prove the activity happened — a file hash, a malicious domain, a registry key. TTPs help you anticipate; IoCs help you confirm.

Malicious Activity Indicators

Beyond malware-specific signs, certain account and resource indicators point to compromise regardless of the tool used:

🔍 Indicators to Watch
  • Account anomalies — lockouts, concurrent sessions, and impossible travel (logins from two distant places too close in time).
  • Resource consumption — abnormal CPU, memory, or bandwidth use, or resources suddenly inaccessible.
  • Out-of-cycle or missing logging — activity at odd hours, or logs that have been cleared to hide tracks.
  • Blocked content and published data — repeated policy blocks, or your data appearing on a leak site.
📌 Example
An endpoint suddenly maxes its CPU and begins encrypting files across a share at high speed, while its logs for the prior hour are missing. Individually, high CPU is normal and a log gap is suspicious — together, they are a confident ransomware indicator, and the EDR isolates the host.

Topic 13B · Physical and Network Attack Indicators

From the cabling to the wire to the air

🔑 Key Idea
Not every attack is software. This topic spans the physical layer, the network, and the wireless airspace — and the network attacks usually unfold as a lifecycle, where each stage leaves its own indicators.

Physical Attacks

Directed at cabling, hardware, or the facility itself: brute-force damage and lock forcing (theft or tampering), environmental attacks (cutting power, network, or cooling), and RFID cloning or skimming of access badges. NFC is a short-range relative of RFID.

🔍 Indicators to Watch

Signs of forced entry on tamper-evident seals, rogue devices added to the premises, and — for cloned badges — the same card used in impossible-travel or concurrent locations.

The Network Attack Lifecycle

1

Reconnaissance

Scanning for hosts, open ports, and software versions. Noisy, but hard to tell from benign scanning.

2

Credential Harvesting

Stealing passwords or secrets to gain authenticated access.

3

Weaponization & Breach

Delivering malicious code to gain a foothold without authenticating.

4

Command & Control

Beaconing out to operate the host remotely, disguised as normal HTTPS.

5

Lateral Movement

Spreading host to host and escalating privileges across the network.

6

Exfiltration

Copying data out — sometimes in small amounts to stay under the radar.

Denial of Service

A DDoS uses a botnet to overwhelm a target by consuming bandwidth or exhausting resources (for example, a SYN flood). Reflected and amplified attacks spoof the victim's address so third-party servers (DNS, NTP, CLDAP, memcached) flood it with large responses.

🔍 Indicators to Watch

Traffic spikes with no legitimate explanation, and services unresponsive or facing abnormally high request volumes. Mitigation relies on high-availability services more than blocking spoofed sources.

Interception: On-Path and DNS Attacks

🕴️

On-Path (AitM)

The attacker sits between two hosts, relaying and possibly altering traffic. ARP poisoning is the classic local-network method.

Tell: floods of gratuitous ARP replies
🧭

DNS Attacks

Poisoning the name-to-IP lookup — via the HOSTS file, server cache, or a local on-path attack — to redirect victims.

Tell: suspect HOSTS entries, DNS log anomalies

Wireless Attacks

Rogue access points and evil twins imitate legitimate networks; wireless DoS jams signals or sends spoofed disassociation frames; KRACK replays the WPA/WPA2 handshake to recover keys.

🔍 Indicators to Watch

Unknown or duplicate access point MAC addresses, typosquatted SSIDs, and APs not connected through expected switches.

Credential Attacks

AttackHow It WorksIndicator
Online passwordGuessing against a live login service.Repeated failed logins, then a success.
Offline / brute forceCracking a stolen hash database away from the system.Audit log of access to SAM, NTDS.DIT, or /etc/shadow.
Password sprayingA few common passwords tried against many accounts.One or two failures spread across many users.
Credential replay (PtH / PtT)Reusing stolen hashes or forged Kerberos tickets (golden/silver).Anomalous logons; LSASS access; tools on host.

Cryptographic Attacks

These prey on weak cryptography: downgrade attacks force weaker protocols (and Kerberoasting forces weak RC4 tickets), while collision and birthday attacks exploit weak hashes to forge signatures.

🔍 Indicators to Watch

Evidence of protocol downgrades in server logs and IDS alerts; unexpected use of deprecated ciphers.

Malicious Code Indicators

⚠ On the Compromised Host
Once code is running, watch for shellcode downloading more tools, credential dumping (SAM, lsass.exe, DCSync), lateral movement via PsExec or PowerShell, and persistence through AutoRun registry keys, scheduled tasks, or WMI event subscriptions.

Topic 13C · Application Attack Indicators

Where the web meets the user — and the logs that catch it

🔑 Key Idea
Web applications expose many interfaces to the public. Attackers exploit the server (injection) and the browser session (hijacking and forgery). These attacks are hard to see in the moment, so detection starts with URL analysis and the web server access log.

Attacks Against the Application

⬆️

Privilege Escalation

Running code at a higher level (vertical) or as another user (horizontal). The clearest indicator of an application attack.

Tell: privilege-use entries in audit logs
💥

Buffer Overflow

Overfilling a memory buffer to overwrite a return address and run attacker code. ASLR and DEP defend against it.

Tell: frequent process crashes
🍪

Replay (Session)

Stealing or guessing a session cookie to hijack an authenticated session.

Tell: same session token from two sources
🎭

Forgery (CSRF / SSRF)

CSRF rides a user's session to act in their name; SSRF makes the server itself issue attacker requests with its own privilege.

Tell: unexpected internal requests

Injection Attacks

All share one flaw — the application trusting unsanitized input. SQL injection and XSS were covered earlier; this lesson adds XML/XXE (reading server files) and LDAP injection (bypassing authentication with a crafted filter). Directory traversal climbs out of the web root with ../ sequences, often hidden with percent encoding, and command injection runs OS shell commands.

🔍 Indicators to Watch

Input containing SQL or script syntax, traversal sequences or percent-encoded paths in URLs, and application errors or crashes that spike when an endpoint is probed.

URL Analysis

A URL is both a destination and a payload. Data is passed after a ? as name-value pairs, and percent encoding — its legitimate job is encoding reserved characters — can be abused to disguise malicious input. Reading the URL is the first detection step.

Web Server Logs

The access log turns a stream of requests into a record you can read. Status codes tell much of the story:

CodeMeaningWhat It May Indicate
4xxClient-side errorsProbing and malformed requests.
403ForbiddenRepeated attempts to reach unauthorized resources.
404 (many)Not FoundA scanner enumerating directories to map the attack surface.
5xx / 502Server-side errorsAttacks causing faults, or a blocked upstream server.
📌 Example
An access log shows one client requesting a normal page, then the same source generating dozens of 404s in seconds as it tries common directory and file names. That burst of 404s is the signature of a tool like Nikto mapping the application — reconnaissance, caught in the log.
📝 Final Lesson Takeaway
Every attack in this lesson — from a virus to a buffer overflow — leaves indicators. The analyst's craft is knowing which traces belong to which attack and correlating them, because one signal is rarely proof. Malware shows on the host, network attacks distort the traffic, credential attacks bend the login patterns, and application attacks surface in the URL and the access log. Put it all together, and you can name the threat with confidence.
